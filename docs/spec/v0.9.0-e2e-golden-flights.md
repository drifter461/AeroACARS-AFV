## Kurzfassung

Die 10 Golden-Flights decken zusammen den vollen v0.9.0-Featureumfang ab:

- **G1** validiert den Standardfall (90-Min-Linienflug ohne Besonderheiten) — Hauptkennwert ist Delta-Sampling-Effizienz und PIREP-Vollständigkeit
- **G2–G4** validieren die drei v0.9.0-Hauptfeatures (Per-Bounce, RTO, GO_AROUND) je in einem dedizierten Szenario
- **G5 + G6** validieren Resilienz unter Sim-Stress (Pause/Disconnect, FPS-Drops)
- **G7** validiert Backward-Compat (alte PIREPs in neuer Webapp)
- **G8 + G9** validieren Degradation bei externen Ausfällen (GlitchTip-offline, Discord nicht installiert)
- **G10** validiert die vollständige Opt-Out-Garantie (jedes neue Feature einzeln abschaltbar)

Jedes Szenario hat: Reproduktions-Anweisung, Fixture-Pfad, erwartete Phase-Sequenz, PIREP-Feld-Tabelle, Bounce-Erwartungen, GlitchTip- und Discord-Verhalten, Privacy-Check und Pass/Fail-Checkliste. Total-QS-Budget am Dokument-Ende.

Out of scope: Performance-Benchmarks (sind eigene Spec), Multi-Pilot-Race-Conditions auf dem VPS (eigene Spec), Webapp-UI-Snapshot-Tests (separater Visual-Regression-Lauf).

## Wie ich diese Tests durchführe

### Setup-Voraussetzungen pro Test-Maschine

Vor dem ersten G-Test muss die Test-Maschine folgenden Zustand haben:

```bash

#    (aus GitHub Actions Artifact des aktuellen claude/*-Branches)
$ aeroacars-cli --version
AeroACARS 0.9.0-rc.N (commit abc1234)

# 2. Recorder-VPS live.kant.ovh erreichbar
$ curl -fsS https://live.kant.ovh/healthz
{"status":"ok","version":"0.9.0-rc.N"}

# 3. GlitchTip-Instanz hat eine dedizierte Test-Org "aeroacars-qs"
#    DSN ist in test-config.json hinterlegt (NICHT der Production-DSN)
$ jq '.error_reporting.dsn' test-config.json
"https://abc@tip.kant.ovh/2"

# 4. Discord-Client läuft (für RPC-Tests) ODER ist deinstalliert (G9)
$ pidof Discord && echo "running" || echo "absent"

# 5. Sim-Auswahl steht vorab fest (MSFS 2024 standard, X-Plane 12 als Backup)
#    Aircraft-Auswahl in der Test-Tabelle pro G-Test angegeben.
```

# G1 — Normaler 90-Min-Flug A320 EDDB→EDDM

**Test-Pilot:** Adrian
**Sim:** MSFS 2024
**Aircraft:** Fenix A320 (mit AeroACARS-Cockpit-State-Beta aktiv)
**Route:** EDDB (Berlin Brandenburg) → EDDM (München Franz Josef Strauß)
**Hauptzweck:** Validierung des „Happy Path" — Delta-Sampling-Effizienz, vollständige Phase-Sequenz, sauberer PIREP ohne neue Edge-Case-Felder, keine Sentry-Events, korrekter Discord-RPC.

## Reproduktions-Anweisung

### Sim-Setup (vor dem Start)

```
1. MSFS 2024 starten, Free Flight Mode
2. Aircraft: Fenix A320, Livery „Lufthansa D-AIQT" (oder GSG-Default)
3. Departure: EDDB Gate „Apron 1 A12" (Cold & Dark)
4. Arrival: EDDM (für Sim-Auto-Setup)
5. Wetter: Live-Weather DEAKTIVIEREN, manuell „Few Clouds, 8 kt 240°, OAT 12°C, QNH 1018"
   (Reproduzierbarkeit — Live-Weather macht jeden Lauf anders)
6. Time: Local Noon (Sim-Time 12:00 UTC)
7. Failures: NONE (kein zufälliger Engine-Failure mitten im Test)
8. Realism: alles auf Standard, KEINE Slew-Mode-Tasten bind
```

### AeroACARS-Setup

```
1. Pilot-Client v0.9.0-rc.N starten
2. Bid auswählen: GSG3184 (EDDB→EDDM, Airbus A320)
3. In Settings: ALLE v0.9.0-Features auf Default (= Delta=ein, Phases=ein, Per-Bounce=ein,
   GlitchTip=AUS — wir aktivieren ihn manuell, Discord=AUS analog)
4. Diese beiden Toggles für G1 manuell EIN:
   - error_reporting.enabled = true (mit Test-DSN aus test-config.json)
   - integrations.discord_rpc.enabled = true
5. MQTT-Capture starten:
   $ mosquitto_sub -h live.kant.ovh -p 8883 --capath /etc/ssl/certs \
     -t 'aeroacars/gsg/+/#' -F '%I %t %p' > results/G1/run-$(date +%F)/mqtt.jsonl &
6. Wireshark starten (für DS1-Privacy-Check), Capture-Filter `host tip.kant.ovh`
7. „Flight Start" im Pilot-Client klicken
```

## Erwartete Phase-Sequence

```
PREFLIGHT      (0:00)    → BOARDING       (2:00)    → PUSHBACK       (12:00)
→ TAXI_OUT     (16:00)   → TAKEOFF_ROLL   (22:30)   → TAKEOFF        (22:45)
→ CLIMB        (23:15)   → CRUISE         (42:00)   → DESCENT        (65:00)
→ APPROACH     (78:00)   → FINAL          (84:00)   → LANDING        (87:00)
→ ARRIVED      (87:45)   → TAXI_IN        (89:00)   → BLOCKS_ON      (91:30)
→ DEBOARDING   (92:00)
```

**Negativ-Erwartung:** weder `REJECTED_TAKE_OFF` noch `GO_AROUND` im Stream. Wenn sie auftauchen → Phase-StateMachine ist false-positive, Fail.

**Timing-Toleranz:** ±60 Sekunden pro Übergang (Pilot ist Mensch, nicht Roboter). Längere Drifts sind ein Sim-Setup-Problem, kein Bug.

## Privacy-Check

**Wireshark-Capture-Auswertung:** Alle HTTPS-POSTs an `tip.kant.ovh` müssen folgende Felder NICHT enthalten:

| Verbotenes Feld | Hintergrund | Wie geprüft |
|---|---|---|
| `pilot_email` | DSGVO | `grep -i 'email' decrypted.json` → 0 Matches |
| `phpvms_api_token` | Auth-Secret | `grep -i 'token' decrypted.json` → 0 Matches |
| `phpvms_password` | Auth-Secret | `grep -i 'password' decrypted.json` → 0 Matches |
| `raw_pilot_id` (= un-gehashter Pilot-ID) | DSGVO | `grep '"pilot_id"' decrypted.json` → leer (nur `pilot_id_hash` erlaubt) |
| PIREP-Notes-Volltext | u.a. potenziell PII | `grep 'PIREP-NOTES' decrypted.json` → 0 Matches |
| Pilot-Klartext-Name | DSGVO | `grep -i 'firstname\|lastname' decrypted.json` → 0 Matches |
| IP-Adresse des Piloten | DSGVO (`server_name` reicht) | Sentry-Default ist mit `send_default_pii=false` deaktiviert — verifizieren |

**Erlaubte Tags (Positivliste):** `va_code`, `pilot_id_hash`, `callsign`, `aircraft_icao`, `simulator`, `client_version`, `phase` (zum Crash-Zeitpunkt), `route` (= `dep-arr`-String).

**G1-spezifisch:** Da 0 Events erwartet werden, ist der Wireshark-Capture trivially „leer" (außer DNS/TLS-Handshake). Wenn doch ein Event da ist, gilt die Negativliste oben.

## Pass/Fail-Kriterien

- [ ] Phase-Sequenz vollständig in MQTT-Capture (= alle 14 Phase-Transitions sichtbar, keine fehlend, keine zusätzlich)
- [ ] Phase-Timing innerhalb ±60 s Toleranz pro Übergang
- [ ] PIREP-JSON validiert mit jq-Skript oben (Exit 0)
- [ ] `touchdown.bounces == null` oder leer (KEIN Array mit Einträgen)
