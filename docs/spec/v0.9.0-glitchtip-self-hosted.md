# v0.9.0 — GlitchTip self-hosted für AeroACARS

**Status:** PROPOSAL — bereit für Implementation nach v0.8.4
**Datum:** 2026-05-18
**Quelle:** Ops-Pain der v0.8.3-Bug-Triage-Woche (3 Bug-Reports, je 1-3h Diagnose via Discord-Screenshots)
**Ziel:** Bug-Reports von Pilots landen automatisch in einer zentralen Triage-Inbox mit Stacktrace + Kontext, statt im Discord als Screenshots gefangen zu bleiben. Self-hosted auf live.kant.ovh (DSGVO-konform, kein Cloud-Lock-in, ~3€/Monat Mehrkosten = $0).

## Pflicht-Vorgabe (Telemetry-Contract)

Diese Spec ist **Konsument** von [`v0.9.0-telemetry-contract.md`](v0.9.0-telemetry-contract.md). Bei Konflikt zwischen dieser Spec und dem Contract gewinnt der Contract.

Konkrete Vertraege die diese Spec einhaelt:
- **Tag-Quellen** (Contract Sektion 1.1 + 1.2): Sentry-Tags `callsign`, `aircraft`, `dep`, `arr`, `simulator`, `phase` werden aus kanonischen Feldern gelesen — KEINE eigenstaendige Berechnung
- **Ownership** (Contract Sektion 3): GlitchTip-SDK schreibt KEINE AeroACARS-Telemetrie-Felder, nur Sentry-eigene Event-Felder
- **Anonymisierung** (Contract Sektion 9 + Spec Sektion „LE5"): `pilot_id` als SHA256-Hash, KEINE Email/Token/PIREP-Notes
- **Feature-Flag** (Contract Sektion 7): `error_reporting.enabled` (Default `false`, Opt-In via First-Run-Banner)
- **Bau-Reihenfolge** (Contract Sektion 6): VPS-Setup (Step 0a) parallel zu Step 0-3, SDK-Integration (Step 4) NACH Steps 1-3 damit Tags stabile Felder nutzen
- **Datenschutz-Gates** (Contract Sektion 9): DS1, DS4, DS5, DS6, DS7 sind GlitchTip-spezifisch und Release-Block bei Fail
- **E2E-Tests** (Contract Sektion 8): G8 (GlitchTip offline) + G10 (Opt-Out) testen Robustheit

## Kurzfassung

Heute landen Pilot-Bugs **ausschließlich** in Discord-Screenshots. Wir raten den Stack-Trace aus dem Pilot-Bildschirmfoto, fragen nach Repro-Steps, durchsuchen VPS-Logs händisch. Eine Diagnose dauert 1-3h pro Report. Bei wachsender Pilot-Anzahl (12 → 30) skaliert das nicht.

**GlitchTip** ist ein Open-Source-Klon von Sentry — gleich API-kompatibel (= Sentry-SDK in allen drei Code-Bases funktioniert 1:1), aber drastisch leichtgewichtiger (5 Container statt 25, 500 MB RAM statt 16 GB, einfache Updates). Wir hosten es selbst auf live.kant.ovh.

Konkreter Mehrwert nach Deployment:
- **Pilot crasht/erhält Error** → 1 Sekunde später bei uns in der Triage-Inbox mit vollem Kontext
- **Bug-Häufigkeit sichtbar** → wir sehen welche Bugs viele Pilots betreffen, nicht nur die lauten Beschwerdeführer
- **Release-Health-Tracking** → wir sehen sofort wenn v0.9.0 schlechter läuft als v0.8.4
- **Custom Messages für Non-Crash-Events** → Adrians „Bounce nicht erkannt"-Klasse wird ebenfalls automatisch gemeldet
- **Discord-Webhook** möglich → neue Bug-Klassen popen automatisch im GSG-Discord

Aufwand: ~3-4 h einmalig (VPS-Setup, SDK-Integration in 3 Code-Bases), danach autonom. Kosten: $0/Mo (selbst gehostet auf vorhandenem VPS).

Out of scope: Performance-Profiling (= Sentry-Premium-Feature), Distributed-Tracing zwischen Services, Mobile-App-SDK (haben wir nicht).

## Problem

### Anchor-Cases der letzten Woche

**Case 1 — Svenny PIREP-Spam (2026-05-17):**
- Symptom: ACARS-Log zeigt 47× rote „PIREP file failed"-Meldungen
- Wie wir es erfahren haben: Discord-Screenshot von Svenny um 18:55
- Wie wir die Ursache fanden: SSH in VPS → DB-Query nach pirep_id → Logs durchsucht → Code gelesen → Hypothese → Pilot rückgefragt
- **Diagnose-Zeit: ~2.5 h**
- Mit GlitchTip wäre das in 2 Min sichtbar gewesen: „Server { status: 400 }: PIREP is read-only — 47 Events, 1 User, betrifft Bonanza TAPA→TRPG, Stack zeigt auf lib.rs:8240"

**Case 2 — Adrian „Bounce nicht erkannt" (2026-05-18):**
- Symptom: Pilot sieht „0 Bounces" obwohl er einen Hopser hatte
- Wie wir es erfahren haben: Discord-Screenshot mit kommentiertem Pfeil
- Wie wir die Ursache fanden: VPS-DB-Query auf touchdowns → JSON-Payload geparst → `bounce_max_agl_ft=14.05` entdeckt → Code-Spec-Konflikt aufgedeckt
- **Diagnose-Zeit: ~1.5 h**
- Mit GlitchTip + Custom Forensik-Events: hätten wir nach 5 ähnlichen Touchdowns gesehen „5 Pilots in 7 Tagen forensic bounce ohne scored bounce" — wir hätten den Bug VOR Adrians Report adressieren können

**Case 3 — Michel Wetlease (2026-05-14):**
- Symptom: Aircraft-Mismatch blockiert Flugstart
- Wie wir es erfahren haben: Discord-Screenshot + freier Text
- Wie wir die Ursache fanden: Code-Lesen + Recorder-DB-Query auf aircraft_aliases
- **Diagnose-Zeit: ~2 h**
- Mit GlitchTip + custom „FlightStartBlocked"-Events: hätten wir die häufigsten Mismatch-Paare als Aggregat gesehen und proaktiv Aliases gepflegt

**Gesamt-Befund:** 3 Bug-Reports in einer Woche = **~6 h reine Diagnose-Zeit**. Mit GlitchTip wären das ~30 Min gewesen (= **−90 % Diagnose-Aufwand**).

### Strukturelle Ursache

Heute haben wir **keinen automatischen Pfad** für Pilot-Errors zurück zu uns:

```
Pilot Error → ACARS-Log (lokal) → Pilot sieht es
            ↓                    ↓
            (verloren wenn       (nur wenn Pilot sich
             Pilot nicht         meldet — 90 % melden
             reagiert)            sich NICHT)
```

Konsequenz: Wir kennen nur die Bugs die laute Pilots melden. Wir kennen **nicht**:
- Wie oft ein Bug auftritt
- Welche Pilots betroffen sind
- Welche Aircraft-Typen / Sims / OS-Versionen betroffen
- Ob ein Bug mit einer bestimmten Version korreliert
- Ob ein „gefixter" Bug wirklich gefixt ist oder weiter auftritt

### Skalierungs-Problem

Heute: 12 Pilots, ~3 Reports/Woche = handhabbar via Discord.

Bei 30 Pilots (Wachstumsziel): ~8 Reports/Woche × 2h = 16h/Woche Bug-Triage. Das ist 1 Arbeitstag pro Woche **nur für Bug-Triage** — nicht skalierbar.

GlitchTip löst das strukturell: Reports kommen aggregiert in einer Triage-Inbox, nicht als 8 verschiedene Discord-Konversationen.

## Leitentscheidungen

### LE1 — GlitchTip statt Sentry-Cloud

| | Sentry Cloud | Sentry Self-Hosted | GlitchTip Self-Hosted (gewählt) |
|---|---|---|---|
| Lizenz | Commercial | BSL → SSPL | MIT |
| Hobby-Preis | $26/Mo | $0 + Server-Ops | $0 + Server-Ops |
| Container-Anzahl | n/a | ~25 | **5** |
| Min RAM | n/a | 16 GB | **512 MB** |
| Update-Aufwand | n/a | komplex | `docker compose pull` |
| SDK-Kompatibilität | nativ | nativ | **1:1 Sentry-SDK** |
| Migrations-Pfad zu Sentry später | Trivial | Trivial | **Trivial** (SDK-kompatibel) |

**Begründung:** Wir bekommen 95 % der Sentry-Features für $0 und 5 % des Ops-Aufwands. Migration zu echtem Sentry später ist eine DSN-Änderung im Code — kein Code-Refactor.

### LE2 — Self-Hosted auf live.kant.ovh

| | Cloud (GlitchTip.com) | Self-Hosted (gewählt) |
|---|---|---|
| Setup | DSN registrieren | docker-compose + Caddy + 30 Min |
| Kosten | $0-15/Mo | $0 |
| DSGVO | EU-Server vorhanden | Volle Kontrolle, alle Daten in EU |
| Datenschutz-Banner | Pilot-Info nötig | Pilot-Info nötig, einfacher zu argumentieren („alle Daten bei uns") |
| Event-Limits | 1k/Mo gratis, dann $ | Unbegrenzt |
| Wartung | keine | `docker compose pull` 1×/Monat |

**Begründung:** VPS hat 3.4 GB RAM frei. Hetzner-Datacenter ist EU (DSGVO-trivial). Kein Vendor-Lock-in. Wartungs-Aufwand vernachlässigbar bei 5 Containern.

### LE3 — Drei SDK-Integrationen: Client (Rust+TS), Recorder (Node), Webapp (TS)

Alle drei Code-Bases bekommen ihr eigenes Sentry-SDK + DSN-Env-Var. Alle senden in dasselbe GlitchTip-Projekt „aeroacars" mit unterschiedlichem `release`-Tag:
- `aeroacars-client@0.9.0` (Rust + React)
- `aeroacars-recorder@0.9.0` (Node)
- `aeroacars-webapp@0.9.0` (React)

In der Triage-Inbox sieht man dann pro Issue von welcher Komponente es kam (= `release`-Tag).

**Begründung:** Komponenten-übergreifende Sichtbarkeit. Bug im Client und Bug im Recorder erscheinen im selben Dashboard, aggregiert nach Komponente.

### LE4 — OPT-IN, Default AUS

**Verbindlicher Default:** `error_reporting.enabled = false`. Bei Fresh-Install + bei Upgrade von v0.8.4 ist der Flag AUS. Es wird KEIN Sentry-SDK initialisiert, kein einziges Event gesendet, solange der Pilot nicht aktiv zustimmt.

Erster App-Start nach v0.9.0-Update zeigt einen einmaligen Hinweis:

> „AeroACARS möchte Fehler- und Crash-Berichte anonymisiert an unseren VPS schicken damit wir Bugs schneller fixen können. Die Daten enthalten keine persönlichen Informationen (kein Email, kein Pilot-ID, kein PIREP-Inhalt). [Aktivieren] [Später entscheiden]"

Pilot wählt:
- **„Aktivieren"** → `error_reporting.enabled = true` persistiert, Sentry-SDK initialisiert, ab sofort Events gesendet
- **„Später entscheiden"** → `error_reporting.enabled` bleibt `false`, kein Banner mehr beim nächsten Start, Pilot kann manuell in Settings aktivieren

Pilot kann jederzeit in Settings widerrufen — sofortige Wirkung (siehe DS7, Contract Sektion 9.1).

**Begründung:**
- **DSGVO Art. 6 (1) a (Einwilligung)** — Pilot stimmt aktiv zu, dokumentiert durch Persistierung des `true`-Werts mit Timestamp im `config.json`
- Art. 6 (1) f (berechtigtes Interesse) wäre theoretisch möglich, ist aber bei Opt-In-Default schwächere Rechtsgrundlage als ausdrückliche Einwilligung
- Pilot kann jederzeit widerrufen ohne Begründung (DSGVO Art. 7 Abs. 3)
- **Konsistenz-Pflicht:** Default-AUS gilt in ALLEN drei Quellen (diese Spec LE4, Telemetry-Contract Sektion 7, Roadmap-Tabelle). Bei Widerspruch zwischen Quellen gewinnt der Telemetry-Contract.

### LE5 — Anonymisierung: KEINE Pilot-ID, KEINE Emails, KEINE Tokens

Sentry-SDK wird so konfiguriert dass:
- `beforeSend(event)` Hook entfernt alle bekannten PII-Felder
- Tokens (phpVMS-API-Key, Bearer-Auth) werden aus Stacks/Breadcrumbs gestrippt
- Pilot-ID wird als anonymisierter Hash übermittelt (= „pilot-23" → SHA256("salt+23") → ersten 8 Zeichen)
- Aircraft-Registration wird übertragen (kein PII)
- Flugnummer wird übertragen (kein PII)
- HTTP-Requests werden auf URL + Status-Code reduziert (= Body wird nicht gesendet)

**Begründung:**
- Pilot-Privacy respektieren
- DSGVO-Argumentation deutlich einfacher
- Verhindert dass versehentlich Login-Tokens im GlitchTip landen

### LE6 — Custom Events für Non-Crash-Fälle

Nicht nur klassische Exceptions, sondern auch unsere bestehenden Activity-Log-WARN/ERROR-Einträge gehen zu GlitchTip via `Sentry.captureMessage()`. Konkret:

- Jeder `log_activity_handle(state, ActivityLevel::Error, ...)` ruft zusätzlich `sentry::capture_message(msg, Level::Error)`
- Jeder `log_activity_handle(state, ActivityLevel::Warn, ...)` mit ausgewählten Codes (z.B. PIREP-Fehler, Aircraft-Mismatch, Auto-Start-Skip) ebenso
- Custom-Events für „interessante" Forensik-Findings (Bounce-Forensik, etc.) — siehe LE7

**Begründung:** „Pilot kann nicht starten" ist die wichtigere Klasse von Errors als „App crasht". Wir wollen Beides sehen.

### LE7 — Forensik-Events nicht in jeder Landung schicken, nur „interessante"

Bei jedem Touchdown ein Forensik-Event zu GlitchTip schicken wäre Daten-Spam. Stattdessen Filter:

- **Hard Landing** (vs < -600 fpm oder peak_g > 1.7): Event mit voller Forensik
- **Light Bounce nur forensic** (= bounce_count=0 aber forensic_bounce_count>0): Event mit Bounce-Details — damit wir Adrian-Class-Issues früh sehen
- **Hard Bounce** (scored_bounce_count >= 2): Event
- **DDS-Violation** (Touchdown vor displaced threshold): Event
- **Wing-Strike-Risiko** (bank > max_bank_landing_deg × 90%): Event

Normale „gute" Landungen erzeugen KEINE Sentry-Events.

**Begründung:** Sentry-Free-Tier hat zwar keine Event-Limits, aber wir wollen die Triage-Inbox nicht zumüllen. Nur „wert dass jemand schaut"-Events.

### LE8 — Release-Tag = AeroACARS-Version, nicht Build-Hash

Jeder Sentry-Event trägt `release: aeroacars-client@0.9.0` (nicht `release: a7b3f9c2`). Damit gruppiert GlitchTip nach offiziellen Versionen, nicht nach individuellen Builds.

**Begründung:**
- Pilots laufen auf offiziellen Versions (= Auto-Updater installiert v0.9.0 etc.)
- Release-Health (= „99.2 % crash-free in v0.9.0") wird damit auswertbar
- Diff zwischen Versionen sichtbar („v0.9.0 hat 12 Bugs, v0.8.4 hatte 8")

### LE9 — Discord-Webhook für neue Issues

GlitchTip kann pro neuer Issue-Class einen Webhook feuern. Wir richten einen GSG-Discord-Channel `#aeroacars-bugs` ein und konfigurieren GlitchTip dorthin.

Trigger:
- Neue Issue-Class (= Bug-Typ den GlitchTip noch nicht gesehen hat)
- Spike (= Bug-Häufigkeit verdoppelt sich innerhalb 1h)
- Regression (= Bug der als „resolved" markiert war taucht wieder auf)

Nicht-Trigger (sonst Spam):
- Jeder einzelne Event eines bekannten Bugs

**Begründung:** Discord-Notifications nur wenn es ETWAS NEUES gibt — keine Doppel-Notifikationen für die gleiche Bug-Klasse.

### Client Rust (`client/src-tauri/src/main.rs`)

```rust
fn main() {
    // Sentry-Init muss VOR Tauri-Setup passieren
    let _guard = sentry::init((
        env!("SENTRY_DSN_CLIENT"),  // = compile-time via build.rs aus env
        sentry::ClientOptions {
            release: sentry::release_name!(),
            // Anonymisierung: alle PII rausfiltern
            before_send: Some(Arc::new(|mut event| {
                // Tokens aus Breadcrumbs strippen
                for crumb in event.breadcrumbs.iter_mut() {
                    if let Some(data) = &mut crumb.data {
                        data.remove("token");
                        data.remove("password");
                        data.remove("authorization");
                    }
                }
                // User-IDs hashen
                if let Some(user) = &mut event.user {
                    user.email = None;  // Email NIE übertragen
                    user.username = None;
                    user.id = user.id.as_ref().map(|id| {
                        let hash = blake3::hash(format!("aeroacars-pii-salt-{}", id).as_bytes());
                        Some(hex::encode(&hash.as_bytes()[..4]))
                    }).flatten();
                }
                Some(event)
            })),
            environment: Some(if cfg!(debug_assertions) { "dev".into() } else { "production".into() }),
            ..Default::default()
        },
    ));

    aeroacars_app_lib::run()
}
```

### Client Rust — Custom Events bei Activity-Log

```rust
// in log_activity_handle (lib.rs:~4350):
fn log_activity_handle(
    app: &AppHandle,
    level: ActivityLevel,
    message: impl Into<String>,
    detail: Option<String>,
) {
    let msg = message.into();
    // Bestehender Pfad zum ACARS-Log-Tab
    log_activity(&app.state::<AppState>(), level, msg.clone(), detail.clone());

    // NEU v0.9.0: bei WARN/ERROR zusätzlich zu Sentry/GlitchTip
    if matches!(level, ActivityLevel::Warn | ActivityLevel::Error) {
        sentry::with_scope(
            |scope| {
                // Kontext-Tags
                if let Some(flight) = app.state::<AppState>().active_flight.lock().unwrap().as_ref() {
                    scope.set_tag("callsign", format_callsign(&flight.airline_icao, &flight.flight_number));
                    scope.set_tag("dep", &flight.dpt_airport);
                    scope.set_tag("arr", &flight.arr_airport);
                    scope.set_tag("aircraft", flight.planned_aircraft_icao.as_deref().unwrap_or("?"));
                }
                if let Some(snap) = current_snapshot(app) {
                    scope.set_tag("simulator", snap.simulator.label());
                }
                if let Some(d) = detail {
                    scope.set_extra("detail", d.into());
                }
            },
            || {
                sentry::capture_message(
                    &msg,
                    if matches!(level, ActivityLevel::Error) {
                        sentry::Level::Error
                    } else {
                        sentry::Level::Warning
                    },
                );
            },
        );
    }
}
```

```tsx
import * as Sentry from "@sentry/react";

Sentry.init({
  dsn: import.meta.env.VITE_SENTRY_DSN_CLIENT_FRONTEND,
  release: `aeroacars-client-frontend@${import.meta.env.VITE_APP_VERSION}`,
  environment: import.meta.env.MODE,
  // Anonymisierung
  beforeSend(event, hint) {
    // Token aus Breadcrumbs strippen
    if (event.breadcrumbs) {
      event.breadcrumbs = event.breadcrumbs.map((crumb) => {
        if (crumb.data) {
          delete crumb.data.token;
          delete crumb.data.authorization;

        return crumb;
      });

    // User-Email NIE
    if (event.user) {
      delete event.user.email;
    }
    return event;
  },
  // Performance-Tracing AUS (Free-Tier-Schoner)
  tracesSampleRate: 0,
  // Replay AUS (Privacy)
  replaysSessionSampleRate: 0,
});
```

```ts
  environment: process.env.NODE_ENV || "production",
  // Recorder ist Server-Side — kein Pilot-PII zu sorgen
  // aber Tokens aus phpVMS-API rausstrippen
  beforeSend(event) {
    if (event.request?.headers) {
      delete event.request.headers["authorization"];
      delete event.request.headers["cookie"];
    }
    return event;
  },
});

// ... rest of recorder startup
```

## Welche Events tracken wir konkret?

### Tag-Allowlist (= das EINZIGE was als Tag/Extra/Context im Event landen darf)

Jeder Sentry-Event MUSS seine Tags aus dieser Allowlist ziehen. Tags die nicht hier stehen werden vom `before_send`-Hook vor dem Senden verworfen. Defense-in-Depth gegen versehentliche PII-Leaks durch zukünftige Code-Änderungen.

| Tag-Name | Typ | Quelle (Owner laut Contract Sektion 3) | Beispiel | Sanitization |
|---|---|---|---|---|
| `os` | string | OS-API | „Windows 11" | — (kein PII) |
| `os.version` | string | OS-API | „10.0.22631" | — |
| `app.version` | string | `client_version` (Build) | „0.9.0" | — |
| `app.component` | string | hardcoded per Codebase | „client-rust" / „client-frontend" / „recorder" / „webapp" | — |
| `simulator` | enum | Sim-Adapter | „msfs" / „xplane" / „other" | — |
| `aircraft` | string | `FlightMeta.aircraft_icao` | „BE36" | nur ICAO-Code, kein Long-Name/Registrierung |
| `airport` | string | Touchdown-Detector `airport` | „TRPG" | nur ICAO-Code |
| `runway` | string\|null | Touchdown-Detector `runway` | „17L" / null | nur Designator |
| `pilot.hash` | string (8 chars) | SHA256(salt+pilot_id)[:8] wenn `anonymize_pilot_id=true` | „a7b3f9c2" | NIEMALS raw pilot_id |
| `pirep.id` | string\|null | `FlightMeta.pirep_id` | „V5rg5e11VWWbJ3J7" | phpVMS-interne ID, kein PII |
| `callsign` | string | `FlightMeta.callsign` | „GSG1824" | — (publik bekannt) |
| `route` | string | `{dep}-{arr}` | „TAPA-TRPG" | — (publik bekannt) |
| `phase` | enum | Contract Sektion 1.3 (18 Werte) | „APPROACH" | — |
| `forensics.version` | string | Touchdown-Detector | „2" | — |
| `error.code` | string | code-internal | „aircraft_mismatch_warning" | — |
| `error.status_code` | int | HTTP-Response | 400 | — |
| `error.kind` | string | classification | „transient" / „hard" / „crash" | — |
| `feature.flag` | string | manual when relevant | „delta_enabled" | — |

**ALLES ANDERE wird gestrippt.** Insbesondere die ursprünglich (= in der ersten Spec-Version) erwähnten `body-snippet`, `query`, `topic_full`, `bid-id`, `reason`, `payload-size`, `tab` — alle wurden zugunsten klarer `error.*`-Tags entfernt. (Das in der Allowlist erlaubte `route` = `{dep}-{arr}` bleibt erlaubt — ICAO-Code-Paar, kein PII.) Details siehe Redaction-Regeln unten.

### Redaction-Regeln pro Event-Klasse (= das was REDIGIERT wird, nie gesendet)

Jede Event-Klasse hat eine kuratierte Wert-Selektion. Hier die explizite Negativ-Liste pro Event-Typ — vom `before_send`-Hook erzwungen:

| Event-Klasse | Felder die WEG müssen (Redaction) | Begründung |
|---|---|---|
| **Panic / Crash** | `panic_payload`, `local_var_values`, `arg_values` in Stack-Frames | können Lat/Lon/Tokens enthalten |
| **PIREP File Failed (transient + hard)** | `response_body`, `request_body`, `headers.authorization`, `headers.cookie`, `url.query` | Response-Body enthält ggf. Pilot-Details; Request-Body enthält PIREP-Notes; Auth-Header enthält Token |
| **Aircraft-Mismatch** | `bid_object`, `sim_snapshot`, `pilot_object` | Vollständige Snapshots können PII enthalten; nur ICAO-Tags reichen |
| **Auto-Start: kein Match** | `bids[]` Array, `lat`/`lon`, exakte `distance_m` | Tags `error.code=no_bid_match` reicht; Distance in groben Buckets (`< 5nm` / `5-50nm` / `> 50nm`) |
| **not_at_departure** | `lat`/`lon`, exakte `distance_nm` | Tag `error.code=not_at_departure` + Distance-Bucket reicht |
| **Forensic-only Bounce** | `samples[]`, `lat`/`lon` | Tag `bounce.max_agl_ft` + `aircraft` reicht; volle Sample-History ist nicht für Bug-Triage relevant |
| **Hard Landing** | `samples[]`, `lat`/`lon`, `aircraft_title` | Tags `vs.fpm`, `peak_g`, `aircraft` (= ICAO) reichen |
| **DDS-Violation** | `lat`/`lon`, `runway_geometry_object` | Tag `airport` + `runway` reicht |
| **SimConnect-Disconnect** | `pipe_path`, `os_error_message_full`, `local_paths` | Pipe-Path kann User-Verzeichnis enthalten; Tag `error.kind=sim_disconnect` + Bucketed-Counter reicht |
| **Webapp React-Errors** | `componentStack` full, `props_object`, `state_object` | Component-Stacks ok, aber Props/State können PIREP-Notes enthalten; nur `error.code=react_error` + Component-Name |
| **Recorder DB-Errors** | `query_full`, `bind_values[]` | Bind-Values können Pilot-IDs etc. enthalten; Tag `error.kind=db_error` + `query.type=SELECT/INSERT/...` (= nur Verb, nicht Statement) |
| **Recorder MQTT-Errors** | `topic_full`, `payload_full`, `payload_size_bytes` | Topic enthält Pilot-ID (= `aeroacars/gsg/23/...`); Tag `error.kind=mqtt_error` + `topic.channel` (= nur „position"/"status"/"phase", nicht Pilot-ID) |

### Updated Event-Klassen-Tabelle (Allowlist-konform)

| Klasse | Code-Stelle | Sentry-Event-Typ | Tag-Set (NUR aus Allowlist) |
|---|---|---|---|
| Panic / Crash | Rust panic-handler | exception | `os`, `os.version`, `app.version`, `app.component`, `simulator`, `aircraft`, `pilot.hash`, `phase` |
| PIREP File Failed (transient) | `lib.rs:8273` | message Warn | `pilot.hash`, `pirep.id`, `error.kind=transient`, `error.code=pirep_file_failed`, `error.status_code` |
| PIREP File Failed (hard) | `lib.rs:8273` | message Error | `pilot.hash`, `pirep.id`, `error.kind=hard`, `error.code=pirep_read_only`/`pirep_validation_failed`/..., `error.status_code` |
| Aircraft-Mismatch | `lib.rs:7755` | message Warn | `aircraft` (= expected ICAO), `app.component=client-rust`, `error.code=aircraft_mismatch`, `simulator` |
| Auto-Start: kein Match | `lib.rs:20666` | message Info | `error.code=no_bid_match`, `airport` (= closest dep), `distance.bucket=lt_5nm/5_50nm/gt_50nm` |

| Forensic-only Bounce | `lib.rs:12862` | message Warn | `aircraft`, `airport`, `runway`, `error.code=forensic_bounce_only`, `forensics.version=2` |
| Hard Landing | `lib.rs` Touchdown-Pfad | message Warn | `aircraft`, `airport`, `runway`, `error.code=hard_landing`, `error.kind=hard_vs`/`hard_g` |

| SimConnect-Disconnect | `lib.rs` Sim-Worker | message Info | `simulator`, `error.kind=sim_disconnect`, `error.code=reconnect_failed`/`pipe_broken`/... |

| Recorder MQTT-Errors | `recorder/mqttSubscriber.ts` | message Error | `app.component=recorder`, `error.kind=mqtt_error`, `error.code=topic_parse_failed`/`payload_invalid`/... |

**Allowlist-Enforcement im Code:**

```rust
// in client/src-tauri/src/sentry_redaction.rs (NEU v0.9.0)
const ALLOWED_TAG_KEYS: &[&str] = &[
    "os", "os.version",
    "app.version", "app.component",
    "simulator", "aircraft", "airport", "runway",
    "pilot.hash", "pirep.id", "callsign", "route", "phase",
    "forensics.version",
    "error.code", "error.status_code", "error.kind",
    "feature.flag",
    "distance.bucket",  // bucketed, not exact
];

pub fn redact_tags(tags: &mut HashMap<String, String>) {
    tags.retain(|k, _| ALLOWED_TAG_KEYS.contains(&k.as_str()));
}
```

Aufgerufen in `before_send`-Hook bevor Event ans Sentry-SDK weitergegeben wird. Defense-in-Depth: selbst wenn ein Entwickler versehentlich `event.set_tag("user.email", ...)` aufruft, würde der Tag rausgestrippt.

## Offene Entscheidungen

1. **Subdomain vs Path?** `tip.kant.ovh` (einfach, separate Logs) vs `live.kant.ovh/glitchtip` (eine Domain) — Vorschlag: Subdomain für Klarheit
2. **Sample-Rate < 100 %?** Bei 30+ Pilots ggf. Performance-Events mit 10% Rate samplen — für jetzt 100 %, später skalieren
3. **Email-Notifications zu uns?** GlitchTip kann auch Email senden zusätzlich zu Discord — Vorschlag: NUR Discord (Email wird Spam)
4. **Pilot-Hash-Salt-Rotation?** Wenn Salt ändert, „verlieren" wir Trend-Daten pro Pilot — Vorschlag: Salt einmalig setzen, nie ändern
5. **Auto-Cleanup-Retention:** GlitchTip kann nach 30/60/90 Tagen alte Events löschen — Vorschlag: 30 Tage (für VPS-Disk-Schoner)
