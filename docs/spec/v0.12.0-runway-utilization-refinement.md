## Ziel

1. **Brems-Disziplin und Aufsetz-Disziplin fair trennen.** Ein Pilot der exzellent bremst, aber innerhalb der normalen Toleranz spät aufsetzt, soll nicht denselben Punktabzug bekommen wie einer der lang ausrollt.
2. **Transparenz** — der Pilot muss in der UI sofort sehen WAS den Score gedrückt hat (Float vs. Ausrollstrecke), nicht raten.
3. **Sicherheits-Aussage bleibt intakt** — das Overrun-Risiko (zu nah am Bahnende) wird weiterhin auf der **echten gesamten** Distanz bewertet, ungekürzt.
4. **Keine stille Score-Drift** — `score_algorithm_version` macht v0.10.0- vs v0.12.0-Scores explizit unterscheidbar; alte PIREPs behalten ihren v2-Score.

## Leitentscheidungen

### LE1 — Float-Toleranz: 15 % der LDA

Die v0.10.0-Formel `distance_used = td_distance + rollout` wird ergänzt um eine **Float-Toleranzzone**. Float bis 15 % der LDA gilt als „normal/akzeptabel" und belastet das Punkte-Banding nicht. Nur Float **über** der Toleranz zählt.

```rust
let float_tolerance_m = 0.15 * lda_m;
let effective_float_m = (td_dist - float_tolerance_m).max(0.0);
let effective_distance_m = rollout + effective_float_m;
```

**Begründung 15 %:**
- Die offizielle Touchdown-Zone (TDZ) reicht bis `min(900 m, LDA/3)` — bei einer typischen 2400–3000 m Bahn sind das ~800–900 m, also ~30 % der LDA. 15 % LDA liegt sicher INNERHALB der TDZ → wer in der erlaubten Zone aufsetzt wird nicht bestraft.
- 15 % ist konsistent mit der `long_float`-Rationale-Schwelle (LE2) — eine Zahl, ein mentales Modell.
- Bei kurzen Bahnen (z. B. 1500 m LDA) sind 15 % nur 225 m — angemessen eng, weil dort jeder Meter zählt. Bei langen Bahnen (3500 m) sind es 525 m — großzügiger, was passt weil dort Float weniger kritisch ist.

**Negativ-Schutz (v0.10.0 LE3 bleibt):** bei `pre_displaced_threshold` ist `td_dist` negativ → `td_dist - tolerance` ist erst recht negativ → `effective_float = 0`. `effective_distance = rollout`. Korrekt — pre-displaced wird weiterhin separat über den Cap auf 55 PT behandelt (v0.10.0 LE3).

**Rundung (R1-P2-2-Fix):** alle internen Berechnungen laufen mit `f32` **ohne Zwischenrundung** — identisch zu v0.10.0 LE4 („Float-Banding, keine Pre-Rundung"). Gerundet wird **ausschließlich fürs Display** (ganze Meter, ganze Prozent). Beispielzahlen in dieser Spec sind durchgehend mit den echten Flight-Log-Werten gerechnet (`td_dist=540,85 m · rollout=442,15 m · LDA=2849,88 m`) und auf 1 Nachkommastelle dargestellt.

### LE2 — `long_float`-Rationale

`long_float` ist explizit der Fall **„Bremsen war exzellent, nur der Aufsetzpunkt war zu spät"**. Der Rationale-Key wird auf `long_float` überschrieben wenn **alle drei** Bedingungen erfüllt sind:

```rust
let float_over_tolerance = td_dist > float_tolerance_m;          // 1. Float über 15 %-Toleranz
let rollout_alone_excellent = (rollout / lda_m) * 100.0 < 30.0;  // 2. Bremsweg ALLEIN wäre excellent
let band_is_good = matches!(final_band, Band::Good);             // 3. finaler Score-Band good/excellent
let long_float = !pre_displaced && float_over_tolerance
                  && rollout_alone_excellent && band_is_good;
```

**R1-P1-1-Fix — warum Bedingung 2 (`rollout / LDA < 30 %`):** R0 prüfte nur `Band::Good`. Aber `Band::Good` umfasst 80 PT UND 100 PT — bei 80 PT ist der Bremsweg nicht zwingend „top". Die Aussage „Bremsweg top" stimmt nur wenn die **Ausrollstrecke für sich allein** ein `excellent_margin`-Niveau erreicht hätte. 30 % LDA ist exakt die `excellent_margin`-Bandgrenze (v0.10.0 LE4). Damit triggert `long_float` nur wenn der Pilot wirklich exzellent gebremst hat und ausschließlich der Float die Punkte gekostet hat.

| Key | DE | EN |
|---|---|---|
| `long_float` | „Bremsweg top — aber spät aufgesetzt" | „Good braking — but touched down long" |

**Warum Bedingung 3 (Band good):** bei finalem Band `ok`/`bad` ist die normale Rationale (`ok_stop` / `long_rollout` / `marginal_runway`) aussagekräftiger — selbst wenn der reine Bremsweg < 30 % war, ist der Gesamt-Eindruck dann nicht mehr „top".

Analogie: v0.10.0 hat `pre_displaced_capped` für „zu früh aufgesetzt". `long_float` ist das Gegenstück „zu spät aufgesetzt". Symmetrisch.

### LE3 — Banding auf `effective_distance`, Overrun-Gate auf echter Distanz

**Kritische Trennung** (Sicherheit vs. Fairness):

```rust
// Overrun-Gate — UNVERÄNDERT auf der ECHTEN, ungekürzten Distanz (v0.10.0 LE4 Step 2)
let raw_ratio_pct = (distance_used / lda_m) * 100.0;   // distance_used = max(td+rollout, rollout)
if raw_ratio_pct > 100.0 {
    return (0, "overrun_risk", Band::Bad);
}

// Banding — NEU auf der toleranzbereinigten Distanz
let effective_ratio_pct = (effective_distance_m / lda_m) * 100.0;
let allowance_pp = if is_heavy { 5.0 } else { 0.0 };
let banding_pct = effective_ratio_pct - allowance_pp;
match banding_pct {
    e if e < 30.0 => (100, "excellent_margin", Band::Good),
    e if e < 50.0 => ( 80, "good_stop",        Band::Good),
    e if e < 70.0 => ( 55, "ok_stop",          Band::Ok),
    e if e < 90.0 => ( 25, "long_rollout",     Band::Bad),
    _             => (  5, "marginal_runway",  Band::Bad),
}
```

**Begründung:** Wer wirklich am Bahnende klebt (echte Distanz > 100 % LDA) bekommt weiterhin `overrun_risk` / 0 PT — die Float-Toleranz darf ein echtes Overrun-Risiko NICHT verstecken. Nur das „Komfort-Banding" (100/80/55/25/5) nutzt die toleranzbereinigte Distanz.

**Konsequenz-Check (BTX8815, echte Flight-Log-Werte):** `td_dist=540,85 m · rollout=442,15 m · LDA=2849,88 m` (A319 = Medium, keine Heavy-Allowance).
- `float_tolerance = 0,15 × 2849,88 = 427,5 m`
- `effective_float = (540,85 − 427,5).max(0) = 113,4 m`
- `effective_distance = 442,15 + 113,4 = 555,5 m`
- `distance_used = (540,85 + 442,15).max(442,15) = 983,0 m`
- `raw_ratio = 983,0 / 2849,88 = 34,5 %` → Overrun-Gate: 34,5 < 100 ✓ kein Overrun
- `effective_ratio = 555,5 / 2849,88 = 19,5 %` → Banding `< 30` → **`excellent_margin`, 100 PT**, `final_band = Good`
- `long_float`-Trigger (LE2): Float 540,85 > Toleranz 427,5 ✓ · `rollout/LDA = 442,15/2849,88 = 15,5 % < 30 %` ✓ · `final_band = Good` ✓ → **`long_float`**
- Ergebnis: **100 PT · Rationale „Bremsweg top — aber spät aufgesetzt"** (vorher v0.11.x: 80 PT „Solider Bremsweg")

### LE4 — Display-Value: ECHTE Bahn-Auslastung, sprachneutral

Die Card-Hauptzeile (`value`) zeigt die **echte** genutzte Distanz + echte Prozent — NICHT die toleranzbereinigte (R0-Frage 1, von Thomas bestätigt: nur echte %, score-relevant gehört in den Erklär-Block). Begründung: das RunwayDiagram zeigt physisch wo der Pilot aufgesetzt + abgerollt ist; eine kleinere Card-Zahl als im Diagramm wäre verwirrend.

**R1-P1-2-Fix — `value` bleibt sprachneutral.** R0 hatte `value` mit deutschem Text (`"35 % der Bahn genutzt · …"`) — das macht den Rust-Crate un-i18n-fähig. R1: `value` ist das **gleiche sprachneutrale Kompakt-Format wie v0.10.0**:

```
value (vom Rust-Crate)  =  "983 m / 2850 m · 35 %"
```

Die Labels (`„genutzt"`, `„von LDA"`, `„der Bahn"`) baut der **TS-Renderer** um den `value` herum — DE/EN/IT über i18n. Damit bleibt `landing-scoring` (Rust) komplett sprachneutral, konsistent mit v0.10.0.

### LE5 — UI: Value-Label + getrennte Extra-Zeilen (TS-gerendert, i18n-fähig)

Aktuelle Card (v0.10.0–v0.11.x):
```
983 m / 2850 m  ·  35 %                       ← value (sprachneutral)
Solider Bremsweg                              ← rationale
▸ davon ~541 m Float vor Aufsetzen            ← extra[0] — hardcoded DE im Rust-Crate
▸ Bahn: LOWS 15, LDA 2850 m                   ← extra[1] — hardcoded DE im Rust-Crate
```

Ziel-Card (v0.12.0):
```
983 m / 2850 m · 35 %     [Label: "35 % der Bahn · 983 m von 2850 m LDA" — TS]
Bremsweg top — aber spät aufgesetzt           ← rationale (i18n long_float)
▸ Aufsetzpunkt: 541 m hinter der Schwelle     ← TS-gerendert aus Record-Feld
▸ Ausrollstrecke ab Aufsetzen: 442 m          ← TS-gerendert aus Record-Feld
▸ Bahn: LOWS 15 · LDA 2850 m                  ← TS-gerendert aus Record-Feld
```

**R1-P1-2 / Frage-5-Fix — `extra` wird NICHT mehr vom Rust-Crate befüllt (für v3-Scores).** R0 wollte die `extra`-Strings im Rust-Crate bauen — das wären hardcoded deutsche Strings (genau das Problem das v0.10.0 schon hatte mit `"davon ~520 m Float…"`). R1-Lösung (Thomas Frage-5-Antwort, Option b):

- `sub_rollout_v2()` setzt für **v0.12.0-Scores `extra = Vec::new()` (leer)**.
- Der **TS-Renderer** baut die 3 Zeilen aus den vorhandenen Record-/Payload-Feldern + i18n-Keys `landing.rollout_extra.*` (DE/EN/IT).

**R2-P1-1-Fix — getrenntes Feld-Mapping (Pilot-Client ≠ Webapp).** Die Datenstrukturen unterscheiden sich:

| Zeile | Pilot-Client (`LandingPanel.tsx`, `LandingRecord`) | Webapp (`LandingAnalysis.tsx`, `FullTouchdownPayload`) |
|---|---|---|
| Aufsetzpunkt | `record.td_distance_from_threshold_m` | `payload.td_distance_from_threshold_m` |
| Ausrollstrecke | `record.rollout_distance_m` | `payload.rollout_distance_m` |
| Bahn-ICAO | `record.runway_match.airport_ident` (nested) | `payload.runway_match_icao` (top-level) |
| Bahn-Ident | `record.runway_match.runway_ident` (nested) | `payload.runway_match_ident` (top-level) |
| LDA-Quelle | `record.runway_match.length_ft` + `record.runway_match.displaced_threshold_ft` (beides ft, nested) | `payload.runway_length_m` + `payload.runway_displaced_threshold_ft` |

**LDA-Berechnung Pilot-Client:** `LDA_m = (length_ft − displaced_threshold_ft) × 0.3048`.
**LDA-Berechnung Webapp:** `LDA_m = runway_length_m − runway_displaced_threshold_ft × 0.3048`.

Beide Renderer kommen zum gleichen LDA-Wert, müssen aber die unterschiedlichen Feld-Pfade kennen. Wenn `runway_match` (Pilot) bzw. `runway_match_icao` (Webapp) fehlt, entfällt die Bahn-Extra-Zeile (die anderen zwei bleiben).

- **Alt-v2-PIREPs** behalten ihre hardcoded-DE `extra`-Strings — Legacy. Der TS-Renderer entscheidet per `score_algorithm_version`: `>= 3` → selbst rendern · `< 3` → gespeicherte `extra`-Strings zeigen.

Damit ist `landing-scoring` (Rust) für v3 komplett sprachneutral, und neue Extra-Zeilen sind i18n-fähig ohne Wire-Format-Änderung.

## Algorithmus (Rust, `sub_rollout_v2` → erweitert)

Der bestehende `sub_rollout_v2()` aus `landing-scoring/src/sub_rollout.rs` wird erweitert. **Kein neuer Funktionsname** — `score_algorithm_version` am Payload macht die Version explizit, die Funktion selbst bleibt die eine SSoT-Compute-Stelle.

```rust
pub fn sub_rollout_v2(input: &RolloutInput) -> SubScoreEntry {
    // Skip-Gate (v0.10.0 LE6) — UNVERÄNDERT
    if let Some(reason) = skip_reason(input) {
        return SubScoreEntry::skipped("rollout", "landing.sub.rollout", reason);
    }
    let td_dist  = input.td_distance_from_threshold_m.unwrap() as f32;
    let rollout  = input.rollout_distance_m.unwrap();
    let runway_m = input.runway_length_m.unwrap();
    let displaced_m = input.runway_displaced_threshold_ft.unwrap_or(0) as f32 * 0.3048;
    let lda_m = runway_m - displaced_m;

    // Echte genutzte Distanz (v0.10.0 LE1 + Negativ-Schutz) — UNVERÄNDERT
    let raw_used = td_dist + rollout;
    let distance_used = raw_used.max(rollout);
    let raw_ratio_pct = (distance_used / lda_m) * 100.0;

    // NEU v0.12.0 (LE1): Float-Toleranz 15 % LDA
    let float_tolerance_m = 0.15 * lda_m;
    let effective_float_m = (td_dist - float_tolerance_m).max(0.0);
    let effective_distance_m = rollout + effective_float_m;
    let effective_ratio_pct = (effective_distance_m / lda_m) * 100.0;

    // Overrun-Check ZUERST, auf der ECHTEN Distanz (v0.10.0 LE4 Step 2) — UNVERÄNDERT
    let (points, rationale, band) = if raw_ratio_pct > 100.0 {
        (0u8, "overrun_risk", Band::Bad)
    } else {
        // NEU v0.12.0 (LE3): Banding auf effective, nicht raw
        let allowance_pp: f32 = if is_heavy(input.aircraft_icao) { 5.0 } else { 0.0 };
        let banding_pct = effective_ratio_pct - allowance_pp;
        match banding_pct {
            e if e < 30.0 => (100, "excellent_margin", Band::Good),
            e if e < 50.0 => ( 80, "good_stop",        Band::Good),
            e if e < 70.0 => ( 55, "ok_stop",          Band::Ok),
            e if e < 90.0 => ( 25, "long_rollout",     Band::Bad),
            _             => (  5, "marginal_runway",  Band::Bad),
        }
    };

    // pre_displaced-Cap (v0.10.0 LE3) — UNVERÄNDERT
    let pre_displaced = input.pre_displaced_threshold.unwrap_or(false);
    let (capped_points, capped_rationale, capped_band) = if pre_displaced {
        let pts = points.min(55);
        let b = if pts >= 75 { Band::Good } else if pts >= 45 { Band::Ok } else { Band::Bad };
        (pts, "pre_displaced_capped", b)
    } else {
        (points, rationale, band)
    };

    // NEU v0.12.0 (LE2, R1-P1-1): long_float-Rationale-Override.
    // DREI Bedingungen — siehe LE2:
    //   1. Float über 15 %-Toleranz
    //   2. Ausrollstrecke ALLEIN waere excellent_margin (rollout/LDA < 30 %)
    //   3. finaler Band ist Good (good_stop / excellent_margin)
    // pre_displaced hat Vorrang (eigener Cap + eigene Rationale).
    let float_over_tolerance = td_dist > float_tolerance_m;
    let rollout_alone_excellent = (rollout / lda_m) * 100.0 < 30.0;
    let (final_points, final_rationale, final_band) = if !pre_displaced
        && float_over_tolerance
        && rollout_alone_excellent
        && matches!(capped_band, Band::Good)
    {
        (capped_points, "long_float", capped_band)
    } else {
        (capped_points, capped_rationale, capped_band)
    };

    let warning = if pre_displaced {
        Some("pre_displaced_threshold".to_string())
    } else {
        None
    };

    // Display-value: SPRACHNEUTRAL (R1-P1-2-Fix), identisches Format wie
    // v0.10.0. ECHTE Auslastung (raw, nicht effective — LE4). Labels macht
    // der TS-Renderer.
    let display_pct = raw_ratio_pct.round() as i32;
    let value = format!(
        "{} m / {} m  ·  {} %",
        distance_used.round() as i32,
        lda_m.round() as i32,
        display_pct,
    );

    // Extra-Zeilen (LE5, R1-P1-2-Fix): v3-Scores lassen `extra` LEER.
    // Der TS-Renderer baut die Float-/Ausrollstrecke-/Bahn-Zeilen aus den
    // ohnehin vorhandenen Record-Feldern + i18n. Kein hardcoded Deutsch
    // mehr im Rust-Crate.
    let extra: Vec<String> = Vec::new();

    SubScoreEntry::scored("rollout", "landing.sub.rollout",
        final_points, value, final_rationale, final_band)
        .with_extra(extra)
        .with_warning(warning)
}
```

**Anmerkung zum `extra`-Feld:** für v0.12.0-Scores ist `extra` bewusst leer (LE5). Das Feld bleibt im Wire-Format (v0.10.0 LE12) — damit Alt-v2-Records ihre gespeicherten `extra`-Strings weiter rendern können. Der TS-Renderer unterscheidet per `score_algorithm_version`.
