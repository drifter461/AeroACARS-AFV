## Ausgangslage (Code-Stand)

`client/src-tauri/crates/sim-xplane/src/dataref.rs` hat eine **statische**
`DATAREFS`-Tabelle: feste Liste `(dataref-name, FieldId)`, die der Adapter per
UDP-RREF abonniert. Es gibt **keinen** per-Flugzeug-Mechanismus. Der Adapter
liefert bereits `aircraft_title` im Snapshot (GSG225: `"Challenger 650 published
by X-Aviation"`) und `aircraft_icao` (`"CL60"`).

---

## Leitentscheidungen

### LE1 — Aircraft-Erkennung über `aircraft_title`

Der X-Plane-Adapter kennt das Flugzeug bereits über `aircraft_title`. Die
Hot-Start CL650 meldet `"Challenger 650 published by X-Aviation"`. Ein Profil
matcht über einen **Substring** des Titels (case-insensitive).

- CL650-Profil-Match: `aircraft_title` enthält `"Challenger 650"` **und**
  `"X-Aviation"` (beides, damit ein Default-X-Plane-Challenger nicht fälschlich
  matcht — der Default trägt nicht „X-Aviation").
- `aircraft_icao` (`"CL60"`) wird **nicht** als Match-Schlüssel genutzt: Default-
  Challenger und Hot-Start-CL650 teilen sich denselben ICAO.

**Offene QS-Frage A-1:** Reicht der Titel-Substring-Match, oder soll der Adapter
zusätzlich „probe-basiert" absichern (ein Override-DataRef anfragen — kommt eine
Antwort, ist es das Study-Add-on)? Vorschlag R0: Titel-Match reicht für v0.12.2,
Probe als Härtung später. Falls Hot-Start den Titel zwischen Versionen ändert,
bricht der Match still — daher LE5 (sauberer Fallback).

### LE2 — Profil-Datenmodell (statisch im Code)

Ein Aircraft-Profil ist eine statische Struktur (analog zur `DATAREFS`-Tabelle —
Pre-1.0 kein config-driven Profil-File):

```rust
struct XplaneAircraftProfile {
    /// Substrings, die ALLE im aircraft_title vorkommen müssen (case-insensitiv).
    title_match: &'static [&'static str],
    /// DataRef-Overrides: ersetzt/ergänzt die Standard-Subscription.
    overrides: &'static [DatarefOverride],
}

struct DatarefOverride {
    field: FieldId,
    dataref: &'static str,
    /// Wie der Rohwert auf die interne Repräsentation abgebildet wird.
    mapping: ValueMapping,
}

enum ValueMapping {
    /// 1:1 übernehmen (z.B. int 0/1 Lights).
    Passthrough,
    /// Detent-Index → interner f32-Wert über eine Lookup-Tabelle.
    DetentTable(&'static [f32]),
}
```

Eine `PROFILES`-Liste hält alle bekannten Profile. Beim Aircraft-Wechsel sucht
der Adapter das erste passende Profil.

### LE3 — CL650-Profil (Hot-Start, X-Plane)

Erfasst von Michel (2026-05-20, DataRefTool):

| FieldId | Override-DataRef | Mapping |
|---|---|---|
| `FlapsHandle` | `abus/CL650/ARINC429/L-DCU-7/words/FCTL/0/FLAPS_LVR` | `DetentTable` (siehe LE4) |
| `BatteryMaster` | `abus/CL650/modules/DC_ELEC/0/wires/BATT_CTRL_PWR` | `Passthrough` (int 0/1) |
| `LightBeacon` | `CL650/overhead/ext_lts/beacon` | `Passthrough` (int 0/1) |
| `LightTaxi` | `CL650/overhead/land_lts/recog_taxi` | `Passthrough` (int 0/1) |

**Nicht im CL650-Profil** (Standard-DataRefs lesen beim CL650 korrekt — am
GSG225-Flugschrieb verifiziert): Gear (`gear/deploy_ratio[0]` = `1.0`),
Triebwerke + Bewegung (saubere Phasenkette), Parkbremse, sämtliche Physik-Werte.
Score-relevant war einzig Flaps.

### LE4 — Flaps-Detent → interner Wert

`FLAPS_LVR` ist ein **Hebel-Detent `int 0–3`** (CL650-Flaps 0 / 1 / 20 / 30),
**kein** Deploy-Ratio. AeroACARS' internes `flaps_position` ist `f32 0.0–1.0`,
und die LANDING-CONFIG-Prüfung (`compute_approach_stability_v2`) nutzt
`flaps_position >= 0.70` als „Landing-Config gesetzt".

**Fachvorgabe:** beim CL650 sind **Flaps 20 (Hebel 2) und Flaps 30 (Hebel 3)**
gültige Lande-Konfigurationen. Die `DetentTable` bildet so ab, dass Hebel 2 + 3
über der 0.70-Schwelle landen:

| Hebel | Flaps | → `flaps_position` |
|---|---|---|
| 0 | 0 | 0.00 |
| 1 | 1 | 0.33 |
| 2 | 20 | 0.85 |
| 3 | 30 | 1.00 |

So bleibt die bestehende `>= 0.70`-Logik in `compute_approach_stability_v2`
**unverändert** — das Profil liefert nur einen passend skalierten Wert. Kein
Score-Logik-Eingriff.

**Offene QS-Frage A-2:** Detent-Tabelle `[0.0, 0.33, 0.85, 1.0]` ok? Insb.: soll
Flaps 20 (Hebel 2) wirklich als „Landing-Config" zählen? (R0-Annahme: ja — CL650
landet regulär mit 20 oder 30.)

### LE5 — Fallback ohne Profil unverändert

Flugzeuge ohne passendes Profil laufen exakt wie heute: Standard-`DATAREFS`-
Tabelle, und falls die Flaps-DataRef dort `0` bleibt, greift die v0.12.1-LANDING-
CONFIG-fail-soft-Behandlung. Kein Verhaltensbruch für Nicht-Profil-Flugzeuge.

### LE6 — Re-Subscription bei Aircraft-Wechsel

Wechselt das Flugzeug zur Laufzeit (Pilot lädt im Sim ein anderes Muster), wertet
der Adapter `aircraft_title` neu aus und abonniert die DataRefs passend neu
(Standard ± Profil-Overrides). Beim Sim-(Re)connect ohnehin Teil des Subscription-
Setups.

## Datenmodell / Wire-Format

**Keine** Wire-Änderung. Das Profil wirkt rein im X-Plane-Adapter; der
`SimSnapshot` (und damit Touchdown-/PIREP-Payload) ist unverändert — `flaps_position`
bleibt `f32 0.0–1.0`, nur mit korrektem Inhalt für Profil-Flugzeuge.
