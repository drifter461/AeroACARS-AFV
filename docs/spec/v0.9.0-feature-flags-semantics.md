## Kurzfassung — warum Kill-Switch-Semantik kritisch ist

Die 8 v0.9.0-Feature-Flags steuern Verhalten, das in zwei Klassen kritisch werden kann:

1. **Externe Side-Effects mit Datenschutz-Auswirkung** — GlitchTip und Discord-RPC senden Daten an Dritt-Systeme (eigener VPS bzw. Discord-Client). Wenn dort ein Bug auftritt (z.B. „Discord-RPC sendet pilot_id ungehasht trotz Anonymisierungs-Flag"), darf das nicht erst beim nächsten App-Release gefixt werden. Pilot oder VA-Admin braucht einen Notfall-Pfad „aus binnen 5 Minuten".
2. **Datenerzeugung mit Konsistenz-Wirkung** — Per-Bounce und Expanded-Phases erzeugen neue Felder in PIREPs. Wenn ein Flag mid-Sprint ausgeschaltet wird (z.B. „GO_AROUND falsch detektiert, deshalb 50 phantomäre go_around_events in der Webapp"), darf das keine Mischzustands-Datenkorruption verursachen. Klar definierte „Disable nach erzeugten Daten"-Semantik schützt den historischen PIREP-Bestand.

Ohne explizite Semantik passieren typische Bugs der Klasse:
- „Pilot hat in Settings GlitchTip ausgeschaltet, aber app sendet weiter weil Settings-Cache stale ist"
- „Per-Bounce-Flag wurde am Tag 3 ausgeschaltet, alte PIREPs zeigen leere bounces[] obwohl Daten da sind"
- „Discord-RPC-Bug am Live-Release-Tag, kein Kill-Switch ausser neuem Release"

Diese Spec schreibt diese Pfade verbindlich vor.

## Allgemeine Prinzipien

### P1. Single Source of Truth pro Flag

Jeder Flag hat **GENAU EINEN Speicherort** und **GENAU EINE Lese-Funktion**. Mehrere Caches sind verboten. Konkret:

- Lese-Funktion: `feature_flag::get<T>(key: &str) -> T` (Rust) bzw. `useFeatureFlag<T>(key)` (React)
- Schreibe-Funktion: `feature_flag::set<T>(key: &str, value: T) -> Result<(), Error>` (Rust)
- Beide Funktionen sind die EINZIGEN Pfade — Komponenten dürfen den Speicherort nicht direkt lesen.
- Begründung: Wenn ein Flag mid-Flight geändert wird, ist garantiert dass der nächste `get` den neuen Wert liefert (= keine Memoization-Inkonsistenz).

### P2. Persistenz-Strategie — `config.json` als zentrale Datei

**Entscheidung:** Wir verwenden die bestehende `config.json` im Tauri-AppData-Pfad (`%APPDATA%/AeroACARS/config.json` unter Windows, `~/Library/Application Support/AeroACARS/config.json` unter macOS). Keine separate `feature-flags.json`.

**Begründung:**
- **Pro `config.json` zentral:** Bestehende Settings-UI liest/schreibt bereits dort (SimBrief-Settings, MQTT-Config). Pilot findet alle Optionen an einer Stelle. Backup ist eine Datei. Migration ist ein Schema-Bump.
- **Contra separate `feature-flags.json`:** Würde Tooling verdoppeln (zwei JSON-Schemas, zwei Validierungen, zwei Backup-Pfade) ohne klaren Nutzen. Power-User wollen einen Editor öffnen, nicht zwei.
- **Power-User-Edit:** Datei ist plain JSON mit Schema-Version. Edit-with-Notepad ist unterstützt (Notfall-Pfad K2 unten).

**Schema-Versionierung:** `config.json` bekommt ein top-level Feld `"schema_version": 2` (v0.8.4 hatte implizit Version 1 ohne expliziten Marker). Beim ersten Start nach v0.9.0-Upgrade läuft ein Migrator (siehe Sektion „Migration-Pfade") der Schema 1 → Schema 2 transformiert und das Backup `config.json.v1.bak` schreibt.

### P3. Migration-Strategie — Conservative Inherit + Explicit New Defaults

Bei Upgrade von v0.8.4 auf v0.9.0:

- **Bestehende Schlüssel** (SimBrief, MQTT, Approach-Stable-Config) werden **verlustfrei übernommen** — keine Defaults überschrieben.
- **Neue Flags** werden mit dem im Contract definierten Default geschrieben. Flag-Werte aus dem 8er-Set sind im Backup nicht vorhanden (= waren in v0.8.4 hartkodiert oder gar nicht existent).
- **Unbekannte Schlüssel** im alten config.json bleiben bei der Migration erhalten (= forward-compat falls jemand ein Feld manuell ergänzt hat).
- **Schema-Bump:** `schema_version` wird von 1 → 2 erhöht NUR wenn die Migration ohne Fehler durchläuft. Bei Migrator-Crash bleibt v1 und v0.9.0-Start scheitert mit klarer Fehlermeldung im Activity-Log + Modal.

### P4. Notfall-Pfade (Kill-Switch-Klassifikation)

Drei Klassen von Kill-Switches mit unterschiedlicher Latenz und Berechtigungs-Anforderung:

| Klasse | Pfad | Latenz | Berechtigung | Use-Case |
|---|---|---|---|---|
| **K1** | Settings-UI-Toggle | <5s wirksam | Pilot selbst | Normaler Opt-out |
| **K2** | `config.json` Edit + App-Restart | 1-2 Min | Pilot selbst (Power-User) | UI-bug oder Pre-Crash-Disable |
| **K3** | VA-Discord-Broadcast „bitte jetzt ausschalten" | 5-30 Min (manuell, Pilot-abhängig) | VA-Admin | Live-Bug-Notfall ohne Remote-Channel |

**Bewusst NICHT enthalten in v0.9.0:** Remote-Push-Channel (z.B. „VA-Admin setzt Flag für alle Pilots gleichzeitig"). Siehe „Out of Scope" — Begründung: Remote-Push würde Auth + Push-Channel-Mgmt + Acknowledgment-Tracking + Conflict-Resolution erfordern, das ist eine Spec für sich. Der manuelle Pfad K3 reicht für die VA-Größe (3-15 aktive Pilots, alle in einem Discord erreichbar).

### P5. Defensive Defaults

Bei Konflikt zwischen „Convenience-für-Pilot" und „Privacy-Risiko" gilt:

- Privacy-relevante Flags (= externes Daten-Pfad: GlitchTip, Discord-RPC) sind **default off**, explizites Pilot-Opt-In.
- Convenience-Flags (= reine Pilot-Client-Verhaltens-Optimierung: Delta-Sampling, Expanded-Phases, Per-Bounce) sind **default on**.
- Anonymisierungs-Flags sind **default on** (= konservative Wahl: lieber zu wenig Tracking als zu viel).

## Pro-Flag-Sektionen

### F1. `position_sampling.delta_enabled` — Delta-Sampling

**Detail-Spec:** `v0.9.0-delta-position-sampling.md`

| Aspekt | Wert |
|---|---|
| **Default Fresh-Install v0.9.0** | `true` |
| **Default bei Upgrade von v0.8.4** | `true` (neuer Key, in v0.8.4 nicht existent) |
| **Persistenzort** | `config.json` → `position_sampling.delta_enabled` |
| **UI-Sichtbarkeit** | Standard-User-Tab „Verbindung & Stream" — sichtbar, kein Power-User-Mode nötig |
| **Remote/Local** | Lokal pro Pilot. Kein Remote-Pfad in v0.9.0 |
| **Wirksamkeits-Latenz** | Sofort (= kein App-Restart). Streamer-Tick liest Flag bei jedem Tick |
| **Warnung im UI** | „Standard ist EIN. Ausschalten erhöht Bandwidth und Recorder-DB-Wachstum um Faktor ~5" |

**Migration bei Upgrade v0.8.4 → v0.9.0:**
- Key existiert in v0.8.4 nicht → wird beim ersten v0.9.0-Start mit `true` geschrieben (= neuer Default)
- Pilot, der explizit v0.8.x-Verhalten will, kann nach Upgrade auf `false` setzen
- Activity-Log beim ersten Start: „Delta-Sampling aktiviert (neu in v0.9.0). Settings → Verbindung & Stream"

**Verhalten bei Disable nach bereits erzeugten Daten:**

| Datenklasse | Was passiert | UI-Sicht Pilot-Client | UI-Sicht Webapp |
|---|---|---|---|
| Bereits gesendete Position-Samples in Recorder-DB | Bleiben unverändert (sparse) | n/a (Stream-Daten) | Live-Map zeigt alte sparse Samples genauso wie vorher |
| Zukünftige Samples | Fixed-Rate-Tick alle 5-8s | n/a | Live-Map dichter |
| Historische PIREPs | unverändert | unverändert | unverändert |

→ **Daten werden nicht gelöscht.** Mischzustand (= sparse historische + dense neue Samples nach Disable) ist akzeptabel weil die Position-Stream-Daten temporär sind (30 Tage Retention).

**Kill-Switch-Pfad:**
- **K1 (UI):** Settings → Verbindung & Stream → Toggle „Delta-Sampling". Sofort wirksam, kein Restart
- **K2 (config.json):** `position_sampling.delta_enabled = false`. App-Restart nicht zwingend, aber empfohlen damit der Streamer den Wert neu lädt
- **K3 (Notfall):** „Alle Pilots: bitte in Settings Delta-Sampling ausschalten" — typisch wenn Recorder-DB-Schema-Mismatch im Server auftritt

### F6. `error_reporting.enabled` — GlitchTip Error-Reporting

**Detail-Spec:** `v0.9.0-glitchtip-self-hosted.md`

| Aspekt | Wert |
|---|---|
| **Default Fresh-Install v0.9.0** | **`false`** (Opt-in!) |
| **Default bei Upgrade von v0.8.4** | **`false`** — auch wenn Pilot „erfahrener Beta-Tester" ist, kein automatisches Opt-in |
| **Persistenzort** | `config.json` → `error_reporting.enabled` |
| **UI-Sichtbarkeit** | Standard-User-Tab „Datenschutz & Telemetrie" — prominent sichtbar mit ausführlicher Erklärung |
| **Remote/Local** | Lokal pro Pilot |
| **Wirksamkeits-Latenz** | Beim Toggle ON: Sentry-SDK wird sofort initialisiert. Beim Toggle OFF: Sentry-SDK wird sofort deinitialisiert (= laufender In-Memory-Buffer wird verworfen, kein Flush an Server) |
| **Warnung im UI** | „Standard ist AUS. Bei EIN sendest du anonymisierte Crash- und Fehler-Reports an `tip.kant.ovh`. Siehe Datenschutz-Seite für Details" |

**Spezial — Boot-Behavior:**
- Wenn `false`: Sentry-SDK wird NICHT initialisiert. `init_sentry()` wird gar nicht aufgerufen. Keine DSN im Speicher.
- Wenn `true`: Sentry-SDK init mit DSN aus Build-Env (= kompiliert ins Binary, kein Runtime-Secret-Risk).

**Migration bei Upgrade v0.8.4 → v0.9.0:**
- v0.8.4 hatte kein GlitchTip → es gibt nichts zu inherit. Default `false` schützt vor unbeabsichtigtem Opt-In.
- Beim ersten Start nach Upgrade zeigt das Settings-Panel einen **One-Time-Banner**: „Neu in v0.9.0: GlitchTip Crash-Reports. Standard ist AUS. Bei Interesse hier opt-in." Banner dismissed nach Klick oder Settings-Besuch.

**Verhalten bei Disable nach bereits erzeugten Daten:**

| Datenklasse | Was passiert | UI-Sicht Pilot-Client | UI-Sicht GlitchTip-Web |
|---|---|---|---|
| Bereits an Sentry gesendete Events | Bleiben in GlitchTip-Postgres (30 Tage Retention via GlitchTip-Spec) | n/a (Server-Daten) | Admin sieht alte Events weiter |
| In-Memory-Buffer (= noch nicht gesendete Events) | Wird beim Disable VERWORFEN, nicht geflusht | n/a | nichts Neues |
| Zukünftige Crashes | KEIN Sentry-Event. App-eigene Activity-Log-WARN/ERROR-Pfad bleibt unverändert | unverändert | nichts |

**Daten-Löschung auf Server-Seite:** Pilot kann via Datenschutz-Seite + E-Mail an Admin eine Löschanforderung stellen (= DSGVO Art. 17). Kein automatischer Löschpfad in v0.9.0 — manueller VPS-Pfad.

**Kill-Switch-Pfad:**
- **K1:** Settings → Datenschutz & Telemetrie → Toggle „Crash-Reports an GlitchTip senden". **Sofort wirksam — kein App-Restart erforderlich (DS7-Pflicht aus Telemetry-Contract Sektion 9).** Implementations-Anforderung: `sentry::close()` wird beim Toggle-Aus aufgerufen (= Sentry-Rust-SDK unterstützt das ab v0.32, [docs.rs/sentry](https://docs.rs/sentry/latest/sentry/fn.close.html)). Lokaler Sentry-Buffer wird VERWORFEN, nicht nachgeflusht. Beim nächsten Toggle-AN wird Sentry neu initialisiert.
- **K2:** `config.json` Edit: `error_reporting.enabled = false`. App-Restart NICHT zwingend nötig — Settings-Watcher liest den Flag pro Tick neu. Restart als Power-User-Option falls Settings-UI hängt.
- **K3 — Notfall (kritischer Live-Bug):** Discord-Broadcast „bitte Datenschutz & Telemetrie sofort ausschalten" → Pilots klicken Settings → Toggle aus → sofort wirksam (siehe K1). Falls Pilots nicht erreichbar: VPS-seitiger Notfall-Pfad: GlitchTip-Endpoint wird auf 503 gesetzt → SDK-POSTs scheitern → Pilots merken nichts, Daten gehen nicht raus.
- **K3 — extreme Notfall (VPS-Stop):** `systemctl stop glitchtip` auf VPS → alle SDKs scheitern beim POST → Pilots merken nichts (keine UI-Wirkung), aber Daten gehen nicht raus.

**Warum „Sofort wirksam"-Pflicht:** DS7 (Telemetry-Contract Sektion 9) sagt expliziert: „Opt-out funktioniert sofort, ohne App-Restart." Das ist Privacy-Anforderung — Pilot muss zu jedem Zeitpunkt seine Einwilligung widerrufen können und das muss SOFORT wirksam sein. Ein „Restart empfohlen"-Hinweis würde die Datenschutz-Validität des Toggles untergraben.

### F7. `integrations.discord_rpc.enabled` — Discord Rich Presence

**Detail-Spec:** `v0.9.0-discord-rich-presence.md`

| Aspekt | Wert |
|---|---|
| **Default Fresh-Install v0.9.0** | **`false`** (Opt-in!) |
| **Default bei Upgrade von v0.8.4** | **`false`** |
| **Persistenzort** | `config.json` → `integrations.discord_rpc.enabled` |
| **UI-Sichtbarkeit** | Standard-User-Tab „Integrationen" — sichtbar mit „Discord muss installiert sein"-Hinweis |
| **Remote/Local** | Lokal pro Pilot |
| **Wirksamkeits-Latenz** | Beim Toggle ON: DiscordPresenceManager wird sofort initialisiert (= IPC-Connect zu lokalem Discord-Client). Beim Toggle OFF: Discord-Status wird sofort gecleart (binnen 5s, siehe DS8 in Telemetry-Contract Sektion 9) |
| **Warnung im UI** | „Standard ist AUS. Bei EIN zeigt dein Discord-Profil deinen aktuellen Flug-Status. Siehe Datenschutz-Seite für angezeigte Felder" |

**Spezial — Sub-Flag `integrations.discord_rpc.anonymize_callsign`:**
- Default `true` bei Fresh + Upgrade
- Wenn `true`: State-Text zeigt „GSG-Flug" statt „GSG3184"
- Wenn `false`: Voller Callsign sichtbar
- Wird im selben Sub-Bereich des Settings-Tabs gezeigt, nur sichtbar wenn `discord_rpc.enabled = true`

**Migration bei Upgrade v0.8.4 → v0.9.0:**
- Neuer Top-Level-Key `integrations.discord_rpc` → komplettes Sub-Objekt wird mit Defaults geschrieben (= `{ enabled: false, anonymize_callsign: true }`)
- Pilot, der schon einen anderen Discord-Bot nutzt (z.B. SimToolkitPro), wird in Settings darauf hingewiesen: „Hinweis: Aktivierung kann mit anderen RPC-Apps kollidieren — Discord zeigt nur eine Quelle gleichzeitig"

**Verhalten bei Disable nach bereits erzeugten Daten:**

Discord-RPC ist **live-only** (siehe Telemetry-Contract Sektion 4: „Discord-Activity-State: live-only, persistiert nirgendwo"). Daher:

| Datenklasse | Was passiert | UI-Sicht Pilot-Client | UI-Sicht Discord |
|---|---|---|---|
| Aktuell angezeigter Discord-Status | Wird gecleart binnen 5s (= `clear_activity()` Call) | Settings zeigt „Status gecleart" | Profil zeigt keinen RPC-Status mehr |
| Aktivierungs-History | n/a (kein Logging in DB) | n/a | n/a |
| Andere Pilots / VA-Server | n/a — RPC ist Discord-Client-lokal, kein Server-Side-State | n/a | n/a |

→ **Keine Daten-Frage** weil Discord-RPC keine persistenten Daten erzeugt.

**Kill-Switch-Pfad:**
- **K1:** Settings → Integrationen → Toggle „Discord Rich Presence"
- **K2:** `integrations.discord_rpc.enabled = false`. Restart empfohlen wenn DiscordPresenceManager fehlerhaft hängt
- **K3 — extrem schnell wegen IPC-Crash-Risiko:** Discord-RPC ist die **wahrscheinlichste Quelle eines Live-Bug-Notfalls** (third-party IPC zu fremdem Prozess). Notfall-Pfade:
  - Discord-Broadcast „bitte Discord-RPC sofort ausschalten + Restart"
  - **Sub-Flag-Override via VPS-Webhook (NICHT in v0.9.0):** wäre möglich aber bewusst nicht gebaut, siehe „Out of Scope"
  - **Notbehelf falls Pilot nicht erreichbar ist:** Discord-Application-ID auf Discord-Developer-Portal deaktivieren → IPC-Connect schlägt fehl → RPC-Manager logged Error im Activity-Log und macht weiter (kein App-Crash). Effektiv: Status verschwindet bei allen Pilots binnen einem Discord-Restart-Cycle (≤ 60s)

### F8. `error_reporting.anonymize_pilot_id` — Hash-Funktion ein/aus

**Detail-Spec:** `v0.9.0-glitchtip-self-hosted.md` (Privacy-Sektion)

| Aspekt | Wert |
|---|---|
| **Default Fresh-Install v0.9.0** | `true` |
| **Default bei Upgrade von v0.8.4** | `true` (neuer Key) |
| **Persistenzort** | `config.json` → `error_reporting.anonymize_pilot_id` |
| **UI-Sichtbarkeit** | **Power-User-Mode** (= versteckt hinter „Erweiterte Einstellungen" Toggle in Datenschutz-Tab) |
| **Remote/Local** | Lokal pro Pilot |
| **Wirksamkeits-Latenz** | Sofort beim nächsten Sentry-Event (= Tag-Computation liest Flag pro Event) |
| **Warnung im UI** | **„NICHT EMPFOHLEN ZU DEAKTIVIEREN.** Bei AUS wird deine pilot_id ungehasht an GlitchTip gesendet — nur für lokale Tests gegen `localhost:9000`. Auf Produktion (= tip.kant.ovh) bleibt der Hash zwingend." |

**Spezial — Server-Side-Enforcement:**
- Auch wenn ein Pilot diesen Flag `false` setzt, wird das Sentry-Backend einen ungehashten pilot_id-Tag **droppen** (= Pre-Processor-Hook in GlitchTip-Config). Das ist die Defense-in-Depth-Schicht. Der Client-Flag ist also nur für Debug-Sessions relevant — kein Privacy-Loophole.
- Begründung für Server-Drop: ungehashte pilot_id ist DSGVO-relevant, Client-Bug darf keine Daten leaken

**Migration bei Upgrade v0.8.4 → v0.9.0:** Neuer Key → Default `true`. Power-User-Bereich ist beim ersten Start kollabiert (= Pilot sieht den Toggle gar nicht).

**Verhalten bei Disable nach bereits erzeugten Daten:**

| Datenklasse | Was passiert | UI-Sicht | Server-Sicht |
|---|---|---|---|
| Bereits gesendete Events (gehashte pilot_id) | Bleiben in GlitchTip | n/a | Admin sieht weiter den Hash |
| Zukünftige Events nach Disable + ON GlitchTip | Client würde ungehashte pilot_id senden — Server droppt sie | n/a | Tag fehlt schlicht im Event |

→ **Kein Daten-Konsistenz-Problem.** Hash und Klar-Form sind unterschiedliche Tag-Werte (= Sentry-Filterung muss beide kennen), aber das ist eine Search-Convenience-Sache, kein Bug.

**Kill-Switch-Pfad:**
- **K1:** Settings → Datenschutz → „Erweiterte Einstellungen anzeigen" → Toggle „pilot_id anonymisieren"
- **K2:** `error_reporting.anonymize_pilot_id = false`
- **K3:** Kein realer Notfall-Pfad denkbar — Anonymisierung schützt den Pilot, nicht den Server. Bei Bug („Hash-Funktion crasht") wird F6 (`error_reporting.enabled`) komplett geknockt

### Was passiert bei Migration-Fehler?

| Fehler | Verhalten |
|---|---|
| Backup-Schreiben scheitert (= Disk-Full) | App-Start abbrechen, Fehler-Modal „Bitte Speicherplatz freimachen oder config.json manuell editieren" |

### Downgrade v0.9.0 → v0.8.4 — was passiert?

**Entscheidung:** v0.9.0-Config wird von v0.8.4 als „mit unbekannten Extra-Keys" gelesen. v0.8.4 hat KEIN Schema-Validierungs-Pfad und ignoriert unbekannte Keys passiv. Konkretes Verhalten:

| Situation | v0.8.4-Verhalten |
|---|---|
| Pilot installiert v0.8.4 über v0.9.0 (= Downgrade) | v0.8.4 liest `config.json`, ignoriert `schema_version=2` und alle 8 neuen Flags. Bestehende Keys (SimBrief, MQTT) funktionieren weiter |
| v0.8.4 schreibt config zurück (= Pilot ändert was in Settings) | v0.8.4 SERIALISIERT NUR die Keys die es kennt → 8 v0.9.0-Flags gehen verloren. `schema_version` geht verloren |
| Pilot installiert wieder v0.9.0 nach Downgrade | Migrator sieht fehlendes `schema_version` → läuft erneut → schreibt 8 Flags mit Defaults. ABER: Pilot-Customizations (z.B. „GlitchTip aktiviert") gehen verloren → werden auf Default zurückgesetzt |

**Begründung dieser Entscheidung:**
- **Pro:** Schema-Versionierung ist trivial (1 Feld), keine forward-compat-Komplexität in v0.8.4 nötig (was wir eh nicht mehr ändern können)
- **Contra:** Pilot-Customizations gehen bei Downgrade+Re-Upgrade verloren
- **Akzeptanz:** Downgrade ist ein Notfall-Pfad. Pilot-Customizations bei einem Notfall-Downgrade zu verlieren ist vertretbar. Wer Downgrade braucht hat eh größere Probleme als Flag-Einstellungen

**Schema-Versioned-Config-Alternative wurde verworfen** weil sie für v0.9.0 overkill ist (würde JSON-Schema-Lib, Validator, Multi-Version-Migrator und Test-Matrix für jede Schema-Version erfordern). Bei v1.0.0 evaluieren wir das Thema erneut wenn die Flag-Anzahl steigt.

## Kill-Switch-Mechanismen — konsolidierte Matrix

| Klasse | Pfad | Wer | Latenz | Verwendbar bei |
|---|---|---|---|---|
| **K1 — Settings-UI** | Toggle in Settings-Panel | Pilot | 0-5s | Normale Opt-out / Opt-in |
| **K2 — config.json Edit** | Datei mit Notepad/VS Code editieren | Pilot (Power-User) | 30-90s (= Restart-Zeit) | UI-Bug verhindert K1 / Vor-Crash-Disable |
| **K3a — VA-Discord-Broadcast** | „bitte ausschalten" + Manual-Mute | VA-Admin | 5-30 Min | Live-Bug, alle Pilots reagieren manuell |
| **K3b — Server-Side-Notbremse** | Sentry-Endpoint 503 / GlitchTip stoppen | VA-Admin auf VPS | <2 Min | GlitchTip-spezifischer Notfall (F6) |
| **K3c — Discord-App-Deaktivieren** | Discord-Developer-Portal | VA-Admin / Thomas | <5 Min, Wirkung ≤ 60s | Discord-RPC-spezifisch (F7) |
| **K4 — neuer Build** | v0.9.0.1 mit hartkodiertem `false` | Maintainer | 24-48 h (Build, Sign, Release) | Letzte Eskalation wenn K1-K3 nicht reichen |

**KEIN Remote-Push-Pfad in v0.9.0.** Begründung:
- Push-Channel würde Auth + Ack + Conflict-Resolution + UI für „Flag wurde remote überschrieben"-Indicator brauchen — eigene Spec
- Für die VA-Größe (3-15 aktive Pilots) reicht K3a (Discord-Broadcast)
- K3b/c decken die zwei wahrscheinlichsten Live-Bug-Klassen (GlitchTip, Discord) server-/third-party-seitig ab

### Notfall-Disable-Playbook „Discord-RPC crasht alle Pilots"

Konkretes Szenario aus dem Briefing („Discord-RPC crasht alle Pilots → wie in 5 Min global ausschalten?"):

```

T+1    Thomas verifiziert lokal → bestätigt
T+2    Discord-Broadcast in #aeroacars-status:

T+5    Pilots die online sind haben K1 ausgeführt (5 min reaktive Latenz)
T+10   Status-Update in #aeroacars-status: „Discord-RPC global gestoppt,
```

```ts
// React-Komponente — KORREKT
const bounces = record.bounces ?? [];
if (bounces.length > 0) {
  return <BounceDetailModal bounces={bounces} />;
}

// FALSCH (Anti-Pattern):
const flagEnabled = useFeatureFlag('forensics.per_bounce_enabled');
if (flagEnabled && record.bounces) { ... }
// → würde alte Daten verstecken sobald Flag aus ist
```

### TypeScript-Seite (`client/src/services/featureFlags.ts`)

```ts
import { invoke } from '@tauri-apps/api/tauri';
import { useEffect, useState } from 'react';

export type FlagKey =
  | 'position_sampling.delta_enabled'
  | 'phases.rejected_take_off_enabled'
  | 'phases.go_around_enabled'
  | 'phases.deboarding_enabled'
  | 'forensics.per_bounce_enabled'
  | 'error_reporting.enabled'
  | 'error_reporting.anonymize_pilot_id'
  | 'integrations.discord_rpc.enabled'
  | 'integrations.discord_rpc.anonymize_callsign';

export async function getFlag(key: FlagKey): Promise<boolean> {
  return invoke<boolean>('feature_flag_get', { key });
}

export async function setFlag(key: FlagKey, value: boolean): Promise<void> {
  return invoke('feature_flag_set', { key, value });
}

export function useFeatureFlag(key: FlagKey): boolean {
  const [value, setValue] = useState<boolean>(false);
  useEffect(() => {
    let cancelled = false;
    getFlag(key).then(v => { if (!cancelled) setValue(v); });
    const unlistenPromise = listen<{key: FlagKey, value: boolean}>(
      'feature-flag-changed',
      (e) => { if (!cancelled && e.payload.key === key) setValue(e.payload.value); }
    );
    return () => {
      cancelled = true;
      unlistenPromise.then(u => u());
    };
  }, [key]);
  return value;
}
```

## Out of Scope

Folgendes ist **bewusst nicht** in v0.9.0:

| Item | Begründung |
|---|---|
| **Remote-Config-Management-Server** | Erfordert Auth + Push-Channel + Ack-Tracking + Conflict-Resolution + UI für „remote-overridden"-Indicator. Eigene Spec (Kandidat für v1.0). VA-Größe (3-15 Pilots) rechtfertigt Aufwand nicht |
| **A/B-Testing-Infrastruktur** | Keine User-Base-Größe die A/B-Tests statistisch sinnvoll macht. Wenn überhaupt, dann später als Cloud-Config-Erweiterung |
| **Feature-Flag-Analytics** | Wir wollen nicht tracken wer welchen Flag wann toggled (= Privacy-bewusste Default-Haltung). Wenn ein Pilot Hilfe braucht, Discord-Frage |
| **Multi-Profile-Support** | Pilot hat genau 1 Config. Test- vs. Prod-Profil-Switching ist nicht relevant |
| **Schema-Versioning > v2** | Solange wir nicht in v1.0 Flag-Konzepte umbauen, reicht ein einmaliger v1→v2-Migrator |
| **Forward-Compat-Hooks** | v0.9.0-Config kann in v0.9.x-Patches additiv erweitert werden (= neue Keys mit serde-default). Schema-Bump erst bei v1.0 |
| **Per-Flag-Audit-Log** | Activity-Log loggt Flag-Wechsel als single-line-info-event. Voll-Audit (= „wann hat wer welchen Flag wie geändert") ist out of scope |
| **Flag-Dependency-Graph** | z.B. „F8 darf nur aktivierbar sein wenn F6 aktiv ist". Aktuell wird Dependency manuell im UI gezeigt (= F7-Sub-Flag-Box ist nur sichtbar bei F7=AN). Generischer Graph ist Overkill |
| **Pilot-Override durch VA via VPS** | Wäre nett für „bitte aktiviert alle Per-Bounce für QS-Sprint", aber siehe Remote-Config-Management oben |
