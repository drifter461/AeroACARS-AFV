## Stream A — Phase-FSM Boarding→Pushback De-Glitch

### A.0 — Befund (verifiziert am BTX8815-Flugschrieb)

`flight-vjmnnNVozNj8MmP7.jsonl.gz`, EDVE-Vorfeld:

| Zeit (UTC) | Ereignis |
|---|---|
| 18:52:53 | `flight_started` — Phase Boarding |
| 18:53:47 → 18:53:58 | **11-Sekunden-Lücke** im Position-Stream (= Sim-Reload/Reposition durch den Piloten) |
| **18:53:58.54** | **EIN** Position-Snapshot mit `groundspeed_kt = 2.63`, `velocity_body_z_fps = -4.43`, AGL springt 9,3→10,5 ft — aber `lat`/`lon` **identisch** zum Sample davor (0,0 m Versatz), `parking_brake = true`, `engines_running = 0` |
| 18:53:58.56 | `phase_changed` Boarding→Pushback **+ `block_off_at` gesetzt** |
| 18:54:04 | nächster Snapshot: `groundspeed_kt = 0.0` wieder |
| 19:27:53 | beide Triebwerke laufen (= echter Pre-Pushback) |
| 19:28:50 | Parkbremse gelöst (= echtes Off-Block) |
| 19:29:15 | `phase_changed` Pushback→TaxiOut |

**Diagnose:** Der Sim-Reload erzeugte eine ~11 s-Lücke im Telemetrie-Stream. Der erste Snapshot danach trug einen transienten Glitch: `groundspeed_kt = 2,63` und eine Rückwärts-Body-Velocity, obwohl das Flugzeug physisch **nicht bewegt** wurde (lat/lon Bit-identisch, Parkbremse gesetzt, Triebwerke aus). Ein einziger Glitch-Tick reichte, um die FSM umzuschalten.

### A.1 — Ursache im Code

`client/src-tauri/src/lib.rs`, `FlightPhase::Boarding`-Branch (~Zeile 15558):

```rust
let on_surface = snap.on_ground
    || (snap.altitude_agl_ft < 5.0 && snap.vertical_speed_fpm.abs() < 50.0);
if on_surface && snap.groundspeed_kt > 0.5 {
    stats.block_off_at = Some(now);
    // … Loadsheet-Logging …
    next_phase = if snap.engines_running == 0 {
        FlightPhase::Pushback
    } else {
        FlightPhase::TaxiOut
    };
}
```

Der Trigger ist **ein einziger Snapshot** mit `groundspeed_kt > 0.5`. Keine Entprellung, kein Abgleich gegen echte Positionsänderung, keine Reload-Lücken-Erkennung. Der Kommentar sagt „actual movement is the only reliable trigger" — aber `groundspeed_kt` aus einem Glitch-Sample ist **keine** echte Bewegung.

**Zweitschaden:** dieselbe Zeile setzt `stats.block_off_at`. Dadurch steht im PIREP die Off-Block-Zeit auf 18:53:58 statt ~19:28:50 — die Blockzeit ist ~35 min zu lang. (`block_snapshot`-Event feuert ebenfalls hier.)

### LE1 — De-Glitch-Gate: echte Bewegung statt Geschwindigkeits-Spike

Der Boarding→Pushback/TaxiOut-Trigger braucht künftig **beide** Bedingungen:

1. **Entprellung** — `groundspeed_kt > 0.5` muss über **≥ 2 aufeinanderfolgende** Snapshots halten. Ein einzelner Spike-Sample zählt nicht.
2. **Echte Positionsverschiebung** — die Großkreis-Distanz zwischen einem beim Boarding-Eintritt (bzw. rollierend) gemerkten Referenzpunkt und der aktuellen Position muss **> `MIN_MOVEMENT_M`** sein (Vorschlag: **5 m**). Ein Glitch-Sample hat 0 m Versatz — er ist damit immun gegen `groundspeed`-Spikes.

Erst wenn beide erfüllt sind, feuert der Übergang und `block_off_at` wird gesetzt.

```rust
// Skizze — finale Form im Code-Slice.
let moved_m = haversine_m(stats.boarding_ref_pos, (snap.lat, snap.lon));
let gs_sustained = snap.groundspeed_kt > 0.5 && stats.prev_gs_over_threshold;
if on_surface && gs_sustained && moved_m > MIN_MOVEMENT_M {
    stats.block_off_at = Some(now);
    next_phase = if snap.engines_running == 0 { Pushback } else { TaxiOut };
}
stats.prev_gs_over_threshold = snap.groundspeed_kt > 0.5;
```

**Warum Positionsverschiebung als Hauptsignal:** ein Reload-Glitch kann `groundspeed_kt` und `velocity_body_*` verfälschen, aber er kann `lat`/`lon` nicht „bewegen" — das Flugzeug steht physisch am Gate. Die Positionsdifferenz ist das fälschungssichere Signal.

**Offene QS-Frage A-1:** `MIN_MOVEMENT_M` = 5 m? Ein echter Pushback bewegt das Flugzeug zügig zweistellig viele Meter; 5 m ist klein genug um den Pushback-START nicht zu verzögern, groß genug gegen GPS-Jitter (typisch < 1 m im Stand).

### LE2 — Reload-Lücken-Erkennung

Wenn die Zeitdifferenz zwischen zwei verarbeiteten Snapshots das erwartete Tick-Intervall **deutlich** übersteigt (Vorschlag: **> 12 s** in Boarding/Pushback, wo das Position-Intervall 4 s ist — siehe `position_interval()`), gilt der erste Snapshot nach der Lücke als **reload-verdächtig**:

- Er löst **keinen** geschwindigkeits-/velocity-abgeleiteten Phasenübergang aus (Boarding→Pushback, TakeoffRoll-Trigger etc.).
- Die **Referenz-Position** (LE1) und `prev_gs_over_threshold` werden auf seine Werte gesetzt/zurückgesetzt — d.h. die Bewegungs-Messung startet sauber nach der Lücke neu.

Damit ist der konkrete BTX8815-Glitch doppelt abgesichert: er kommt direkt nach einer 11 s-Lücke (→ LE2 verwirft ihn) und hätte auch ohne LE2 mangels Positionsverschiebung (→ LE1) nicht ausgelöst.

**Offene QS-Frage A-2:** Schwellwert 12 s, oder lieber an `position_interval(phase)` koppeln (z.B. `> 3×`)? Vorschlag: `> 3 × position_interval` — skaliert automatisch mit der Phase.

**Offene QS-Frage A-3:** Soll LE2 nur Boarding/Pushback absichern, oder die ganze FSM (auch Climb/Descent/Approach)? Ein Reload im Reiseflug könnte ähnliche Glitches erzeugen. Vorschlag R0: **nur Boarding/Pushback** in v0.12.1 (kleiner, gezielter Fix); FSM-weite Reload-Härtung als separater späterer Task.

### LE3 — `block_off_at` / OOOI-Korrektheit

`block_off_at` wird ausschließlich im LE1-Pfad gesetzt. Mit LE1 ist damit automatisch die Off-Block-Zeit korrekt — kein separater Code, nur die Konsequenz von LE1. **Akzeptanz:** im Re-Test darf `block_off_at` nicht mehr auf einem Glitch-Sample liegen.

Kein Re-Score / keine Rückwirkung auf bereits eingereichte PIREPs (forward-only).

## Stream B — VA-Härtung: nur German Sky Group

### B.0 — Befund (GitHub-Issue #16)

Ein VA-fremder Tester betrieb AeroACARS und löste einen Flug aus. Der Issue-Reply nennt die Annahme „domain is hard-coded … other VAs are blocked" — die stimmt für das **offizielle Binary nicht als echte Sperre**:

| Schicht | Heutiger Stand | Lücke |
|---|---|---|
| Client-Login | `ALLOWED_PHPVMS_HOST = "german-sky-group.eu"` (Konstante in `lib.rs:4634`), `phpvms_login` erzwingt den Host. | Quellcode ist **Open-Source (MIT)** — eine Zeile ändern + `tauri build` und der Lock ist weg. |
| Client-Login | Nach `get_profile()` wird die Airline des Piloten **nicht geprüft**. | Jeder Account auf german-sky-group.eu kommt durch — auch ein nicht-GSG-Account auf einer Multi-Airline-phpVMS-Instanz. |
| Recorder `provision.ts` | `validatePhpVmsKey()` akzeptiert **jeden** API-Key, für den `/api/user` ein User-Objekt mit `id` liefert. | `user.airline` / `airline_id` wird geholt, aber **nie geprüft** (Kommentar Zeile 68: „Später: airline.icao verwenden"). |

**Vorgabe von Thomas (QS-Antwort):** „alle GSG-Piloten — keine andere VA — ich möchte keine GSG-Piloten freischalten müssen — nur GSG." → Das Gate ist **auf Airline-Ebene**, kein Pilot-Whitelisting. Wer in phpVMS zur Airline GSG gehört, darf; alle anderen nicht.

### LE4 — Recorder-seitiges Airline-Gate (die echte Durchsetzung)

`recorder/src/provision.ts`: direkt nach `validatePhpVmsKey()` ein Airline-Gate:

```ts
const user = await validatePhpVmsKey(cfg.phpvmsBaseUrl, apiKey);
if (!user) return { error: "phpVMS API key not valid", status: 401 };

// v0.12.1 — VA-Gate: nur German Sky Group.
if (!isGsgAirline(user)) {
  return { error: "not a German Sky Group pilot", status: 403 };
}
```

`isGsgAirline()` prüft `user.airline?.icao` gegen den GSG-Wert; Fallback auf `user.airline_id` falls `airline` nicht expandiert mitkommt. Bei Nicht-GSG: **keine MQTT-Provisionierung**, kein Broker-User, kein Eintrag in `provisioned_pilots`.

**Wichtig — das Gate läuft VOR dem Cache-Lookup** (`getProvisionedPilot`), damit auch ein evtl. früher (vor v0.12.1) provisionierter Nicht-GSG-Pilot beim nächsten Provision-Call abgewiesen wird. Das ist die einzige Schicht, die ein Client-Fork **nicht** umgehen kann — sie läuft auf GSG-Servern.

**Offene QS-Frage B-1:** Sollen bereits in `provisioned_pilots` eingetragene Nicht-GSG-Piloten (falls vorhanden) zusätzlich aktiv `revoked` werden? Vorschlag R0: separater Einmal-Audit per CLI (`listProvisionedPilots` zeigt `phpvms_user_id` → Airline nachschlagen), nicht im Hot-Path. Thomas entscheidet ob nötig.

### LE5 — Client-seitiges Airline-Gate (Defense-in-Depth + saubere UX)

`phpvms_login` in `client/src-tauri/src/lib.rs`: nach `client.get_profile()`:

```rust
let profile = client.get_profile().await?;
if !is_gsg_airline(&profile) {
    return Err(UiError::new(
        "not_gsg",
        "AeroACARS ist exklusiv für Piloten der German Sky Group.",
    ));
}
```

`is_gsg_airline()` prüft `profile.airline.icao` (vorhanden im `api-client::Profile`) gegen den GSG-Wert, Fallback `airline_id`.

**Zweck:** ein Nicht-GSG-Pilot bekommt eine **klare Fehlermeldung** statt eines stummen MQTT-Provisionierungs-Fehlschlags (LE4) — bessere UX. Es ist bewusst **keine** echte Sicherheitsschicht (Client ist Open-Source, umgehbar) — die echte Durchsetzung ist LE4. LE5 ist die erste, ehrliche Linie.

### LE6 — GSG-Identifier als zentrale Konstante

| Ort | Konstante | Wert |
|---|---|---|
| Client `lib.rs` | `ALLOWED_AIRLINE_ICAO` (neben dem bestehenden `ALLOWED_PHPVMS_HOST`) | `?` — QS-R0 |
| Recorder `config.ts` | `cfg.allowedAirlineIcao` (Env-überschreibbar, Default fix) | `?` — QS-R0 |

**Offene QS-Frage B-2 (blockierend):** Wie lautet der GSG-Identifier in phpVMS?
- `airline.icao` — vermutlich `"GSG"` (Pilot-Idents haben die Form `GSG0001`, siehe `api-client::Profile.ident`-Doc). Bitte exakt bestätigen.
- `airline_id` — die Integer-ID der GSG-Airline (Fallback falls `/api/user` das `airline`-Objekt nicht expandiert). Bitte den Integer nennen.

Das Gate prüft primär `icao`, fällt auf `airline_id` zurück. Beide Werte werden in QS-R0 von Thomas fixiert.

**Offene QS-Frage B-3:** Liefert `german-sky-group.eu/api/user` das verschachtelte `airline`-Objekt mit `icao`? Wenn nein, muss das Gate auf `airline_id` laufen — daher braucht es B-2 in jedem Fall. (Verifizieren lässt sich das mit einem echten GSG-API-Key gegen `/api/user`.)

This phpVMS account belongs to a different VA." | „AeroACARS è esclusivo per i piloti della German Sky Group. Questo account phpVMS appartiene a un'altra VA." |

### LE8 — Nicht im Scope (bewusst dokumentiert)

- **Open-Source-Forks gegen fremde phpVMS-Instanzen** sind nicht verhinderbar (MIT-Lizenz) und auch **kein Ziel** — wer AeroACARS gegen seine *eigene* VA-Infrastruktur baut, berührt GSG nicht. Geschützt wird GSG-Infrastruktur (Recorder, MQTT-Broker, live.kant.ovh), nicht der Quellcode.
- **phpVMS-Registrierungs-Policy:** falls german-sky-group.eu offene Selbst-Registrierung erlaubt, kann sich jeder einen GSG-Account anlegen und wäre dann „GSG-Pilot". Das ist eine **phpVMS-Admin-Einstellung** (Registrierung → Admin-Freigabe), kein AeroACARS-Code. Wird hier nur als **Empfehlung** notiert; Thomas entscheidet separat. v0.12.1 setzt das Airline-Gate um — die Account-Vergabe-Policy bleibt GSG-Admin-Sache.

## Datenmodell / Wire-Format

Stream A: **keine** Wire-Änderung. `block_off_at` und die `phase_changed`-Events behalten ihr Format — sie feuern nur zum korrekten Zeitpunkt.

Stream B: **keine** Wire-Änderung. Das Provisioning-Response-Format (`ProvisionResult`) ist unverändert; im Nicht-GSG-Fall kommt der bestehende `{ error, status }`-Pfad mit `status: 403`.

Neue interne Felder in `FlightStats` (Stream A, nicht persistiert / nicht im Wire):
- `boarding_ref_pos: Option<(f64, f64)>` — Referenz-Position für die Bewegungs-Messung.
- `prev_gs_over_threshold: bool` — Entprellungs-Merker.
- `last_snapshot_at: Option<Instant/DateTime>` — für die Lücken-Erkennung (LE2; evtl. existiert schon ein Äquivalent — im Code-Slice prüfen).
