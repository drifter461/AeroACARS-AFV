# AeroACARS v0.8.4 — Hotfix: keine doppelten Stat-Pills mehr unter dem Runway-Diagramm

## 🇩🇪 Deutsch

**v0.8.4 — Reiner Hotfix für einen UI-Bug aus v0.8.2: unter dem Runway-Diagramm wurden zwei Pill-Reihen mit überlappenden Feldern gerendert, teilweise mit widersprüchlichen Werten. Eine Reihe (die alte aus v0.7.x) ist jetzt entfernt.**

### Was war kaputt

Im Landungen-Tab zeigte AeroACARS unter dem Runway-Diagramm zwei Pill-Reihen, die im Wesentlichen dieselben Felder anzeigten — aber teilweise mit unterschiedlichen Werten für denselben Touchdown:

| Feld | obere Pill-Reihe (V2) | untere Pill-Reihe (legacy) |
|---|---|---|
| BAHN-AUSLASTUNG | 52 % | 38 % |
| MITTELLINIE | „auf CL" | „0.3 m Mitte" |
| BAHN | „EVRA/18 (Asphalt)" | „EVRA/18 (ASP)" |

Beide Auslastungs-Werte waren technisch „korrekt", aber unterschiedlich gerechnet:
- **V2 (52 %):** `(Touchdown-Position-hinter-Schwelle + Ausrollstrecke) ÷ Bahnlänge` = wieviel Bahn wurde vom Schwellenkopf bis zum Bremspunkt verbraucht
- **Legacy (38 %):** `Ausrollstrecke ÷ Bahnlänge` = veraltete v1-Formel aus pre-v0.8.0

Pilot Thomas hat das mit einem Fenix-A320-Screenshot von einer EVRA-Landung gemeldet (2026-05-18).

### Warum das passiert ist

Beim v0.8.2-Release (RunwayDiagram v2) wurde die neue V2-Component mit eigenen Pills eingeführt — aber die alte `<dl>`-Liste in `LandingPanel.tsx` wurde nicht entfernt. V2 und Legacy liefen seit v0.8.2 parallel und zeigten dieselben 11 Felder zweimal.

### Was sich ändert

Die legacy `<dl>`-Liste ist komplett entfernt (171 Zeilen weg, 24 hinzugefügt). V2-Pills decken alle Felder ab: Bahn, Länge, Hinter Schwelle, Mittellinie, Ausrollstrecke, Bahn-Auslastung, Navdata-Quelle, TDZ, Aim-Point, TCH, DDS-Warnung.

Bei untrusted geometry (Privatplätze ohne saubere Runway-DB) rendert weder V2 noch die alte Liste — die Trust-Warn-Box oberhalb erklärt warum (vorher zeigte legacy auch da teilweise weiter, was zur V2-Spec inkonsistent war).

### Was Piloten tun müssen

Nichts — Auto-Updater rollt v0.8.4 aus. Beim ersten Klick auf einen Landungs-Eintrag siehst du nur noch **eine** Pill-Reihe mit konsistenten Werten.

## 🇬🇧 English

**v0.8.4 — Pure hotfix for a UI bug from v0.8.2: two pill rows with overlapping fields were being rendered under the runway diagram, sometimes with conflicting values. One row (the legacy one from v0.7.x) is now removed.**

### What was broken

In the Landings tab, AeroACARS rendered two pill rows under the runway diagram showing essentially the same fields — sometimes with different values for the same touchdown:

| Field | Top pill row (V2) | Bottom pill row (legacy) |
|---|---|---|
| RUNWAY USAGE | 52 % | 38 % |
| CENTERLINE | "on CL" | "0.3 m centre" |
| RUNWAY | "EVRA/18 (Asphalt)" | "EVRA/18 (ASP)" |

Both usage values were technically "correct" but calculated differently:
- **V2 (52 %):** `(touchdown-distance-past-threshold + rollout) ÷ runway-length` = how much runway was consumed from threshold to braking point
- **Legacy (38 %):** `rollout ÷ runway-length` = obsolete v1 formula from pre-v0.8.0

Pilot Thomas reported this with a Fenix A320 screenshot from an EVRA landing (2026-05-18).

### Why it happened

At the v0.8.2 release (RunwayDiagram v2), the new V2 component was introduced with its own pills — but the legacy `<dl>` list in `LandingPanel.tsx` was not removed. V2 and legacy ran in parallel since v0.8.2, showing the same 11 fields twice.

### What changes

The legacy `<dl>` list is completely removed (171 LOC removed, 24 added). V2 pills cover every field: runway, length, past threshold, centerline, rollout, runway usage, navdata source, TDZ, aim point, TCH, DDS warning.

For untrusted geometry (private strips without proper runway DB) neither V2 nor the legacy list renders — the trust warning box above explains why (previously legacy also rendered partially in untrusted cases, inconsistent with the V2 spec).

### What pilots need to do

Nothing — auto-updater rolls out v0.8.4. On first click on a landing entry you'll only see **one** pill row with consistent values.
