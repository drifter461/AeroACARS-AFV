## 🇩🇪 Deutsch

**Eine Sinkrate, eine Wahrheit — Schluss mit dem Werte-Dschungel + SimBrief-Daten schon vor dem IFR-Start.**

### Warum dieses Release

Realer Pilot-Fall der das ausgelöst hat: **DAL804** zeigte im phpVMS-PIREP und in AeroSore `-407 fpm`, aber im AeroACARS-Landung-Tab `-364 fpm`. Beide Werte kamen vom Pilot-Client — der eine aus dem MSFS-SimVar `PLANE TOUCHDOWN NORMAL VELOCITY` (latched), der andere aus dem 50-Hz-Buffer-Edge (interpoliert). Der Pilot dachte zu Recht: das passt nicht.

Dieses Release konsolidiert die Sinkrate auf **einen einzigen kanonischen Wert** (50-Hz-Buffer-Edge, `vs_at_edge_fpm`) — und der wird überall gleich angezeigt: Touchdown-Card, Sinkrate-Forensik, phpVMS-PIREP, AeroSore-VPS-Reports.

### Was ist neu

🎯 **Sinkrate-Konsistenz (v0.7.11)**
- Backend: `vs_at_edge_fpm` ist jetzt die kanonische Score-Basis. Wenn die v2-Touchdown-Forensik den primären Wert REJECTed, wird der 50-Hz-Edge-Wert genutzt — nicht mehr der MSFS-SimVar-Fallback
- Pilot-Client Touchdown-Card aufgeräumt: nur noch EINE Sinkrate sichtbar (Landing-Rate), die smoothed-VS-Tiles (250/500/1000/1500 ms) leben jetzt in der Sinkrate-Forensik-Sektion wo sie mit Aufklärungs-Text und Coaching-Tipp Sinn ergeben
- VPS-Webapp: "Algorithmen-Forensik"-Card raus, "Disagreement Δ>50"-Filter raus, Lua-vs-Time-Tier-Schätzer-Vergleich nicht mehr in der Pilot-UI

✨ **Pre-Flight-SimBrief-direct (v0.7.10)**
- Frische SimBrief-OFP-Werte (Fuel, Route, Pax, Wind) erscheinen jetzt schon **vor dem IFR-Start** in der Bid-Liste — direkt von simbrief.com geholt, nicht aus dem phpVMS-Pointer-Cache
- Grünes Banner "✓ Frische SimBrief-Werte geladen" beim Klick auf "Aktualisieren"
- Neuer Tauri-Command `bid_simbrief_preview` ohne Phase-Lock-Gate

🔄 **SimBrief-Callsign-Soft-Warning (v0.7.9)**
- Wenn DEP/ARR matchen aber das OFP-Callsign vom Bid abweicht (typischer Eurowings-Fall: phpVMS `EWL9725` ↔ SimBrief `EWG4PY`), wird der OFP **trotzdem geladen** — vorher Hard-Block
- Gelber Soft-Warning-Notice statt rotem Fehler
- Hard-Block bleibt nur bei echtem DEP- oder ARR-Mismatch

### Konsistenz auch für Alt-Reports

Die VPS-Webapp re-mapped automatisch alle historischen Touchdowns auf den 50-Hz-Edge-Wert (sofern der Buffer-Dump damals erfasst wurde, d.h. ab Pilot-Client v0.5.39+). Du musst nichts machen — beim nächsten Reports-Tab-Aufruf zeigen Alt-Flüge wie DAL804 jetzt den korrekten Wert.

### Hinweis für phpVMS-VA-Owner

Neue PIREPs senden ab sofort den korrekten Edge-Wert als `landing_rate`. Bereits gefilte PIREPs in der phpVMS-Datenbank bleiben auf dem alten Wert — wer Backfill möchte, kann manuell per SQL korrigieren (Anleitung im Forum).

## 🇬🇧 English

**One sink rate, one truth — no more confusing parallel values + SimBrief data already available before IFR start.**

### Why this release

Real pilot case that triggered this: **DAL804** showed `-407 fpm` in the phpVMS PIREP and AeroSore, but `-364 fpm` in the AeroACARS Landung tab. Both values came from the pilot client — one from the MSFS SimVar `PLANE TOUCHDOWN NORMAL VELOCITY` (latched), the other from the 50 Hz buffer edge (interpolated). The pilot rightly thought: this doesn't add up.

This release consolidates the sink rate to **a single canonical value** (50 Hz buffer edge, `vs_at_edge_fpm`) — and it's shown identically everywhere: touchdown card, sink-rate forensics, phpVMS PIREP, AeroSore VPS reports.

### What's new

🎯 **Sink-rate consistency (v0.7.11)**
- Backend: `vs_at_edge_fpm` is now the canonical score basis. When the v2 touchdown forensics REJECT the primary value, the 50 Hz edge value is used — no more falling back to the MSFS SimVar
- Pilot-client touchdown card cleaned up: only ONE sink rate visible (landing rate), the smoothed VS tiles (250/500/1000/1500 ms) now live in the Sink-Rate Forensics section where they make sense with explanation and coaching tip
- VPS web app: "Algorithm Forensics" card removed, "Disagreement Δ>50" filter removed, Lua vs. Time-Tier estimator comparison no longer in the pilot UI

✨ **Pre-flight SimBrief direct (v0.7.10)**
- Fresh SimBrief OFP values (fuel, route, pax, wind) now appear in the bid list **before IFR start** — fetched directly from simbrief.com, not from the phpVMS pointer cache
- Green banner "✓ Fresh SimBrief values loaded" when clicking "Refresh"
- New Tauri command `bid_simbrief_preview` without phase-lock gate

🔄 **SimBrief callsign soft warning (v0.7.9)**
- If DEP/ARR match but the OFP callsign deviates from the bid (typical Eurowings case: phpVMS `EWL9725` ↔ SimBrief `EWG4PY`), the OFP is **still loaded** — previously a hard block
- Yellow soft-warning notice instead of red error
- Hard block remains only for true DEP or ARR mismatches

### Consistency for old reports too

The VPS web app automatically remaps all historical touchdowns to the 50 Hz edge value (provided the buffer dump was captured at the time, i.e. pilot client v0.5.39+). You don't need to do anything — on your next Reports tab visit, old flights like DAL804 will show the corrected value.

### Note for phpVMS VA owners

New PIREPs from now on send the correct edge value as `landing_rate`. Already-filed PIREPs in the phpVMS database remain at the old value — anyone who wants a backfill can correct manually via SQL (instructions in the forum).
