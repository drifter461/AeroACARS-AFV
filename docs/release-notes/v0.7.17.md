## 🇩🇪 Deutsch

**v0.7.17 — Fenix-Polish und Bug-Bündel. Sieben Bugs aus dem Tester-Feedback zu v0.7.16 sind raus. Highlights: Fenix-Profil wird jetzt automatisch erkannt (kein Toggle mehr), SimBrief-Refresh greift endlich beim Flug-Start, und der Auto-Start sagt dir warum er nicht feuert wenn er nicht feuert.**

### Für alle Piloten

#### Fenix wird automatisch erkannt
- Kein Opt-in-Toggle mehr in Settings → Beta-Features
- Sobald AeroACARS eine FenixA319 / A320 / A321 erkennt, greifen die verbesserten Cockpit-LVAR-Mappings (Landing-/Nose-/Wing-Lights, Park-Brake, Signs etc.) automatisch
- Wenn du Fenix fliegst: keine Aktion nötig, läuft einfach

#### SimBrief-Aktualisierung wirkt jetzt vor dem Flug-Start
- Vorher: nach „Aktualisieren" im Bid-Tab + Flug-Start kamen trotzdem die alten OFP-Daten
- Jetzt: Flug-Start holt den **aktuellsten** OFP direkt von simbrief.com (sofern in Settings → SimBrief ein Identifier gesetzt ist)
- Fallback auf den phpVMS-Bid-Pointer wenn der Direct-Fetch fehlschlägt
- Workflow „SimBrief neu generieren → Aktualisieren → Flug starten" funktioniert jetzt wie erwartet

#### Bahn-Auslastung-Score endlich aircraft-aware
- A321 auf 3 km Bahn mit 2 km Rollout? Vorher: 25/100 („sehr lang ausgerollt"). Jetzt: 80/100 („good_stop")
- Drei Aircraft-Kategorien mit eigenen Schwellen: Light-GA / Medium-Narrowbody / Heavy-Widebody
- Cessna-Piloten merken keine Änderung — Schwellen identisch zu v0.7.16

#### Auto-Start: jetzt mit Diagnose
- Wenn Auto-Start nicht feuert, steht der Grund im Activity-Log:
  - X-Plane: „Aircraft-Titel fehlt → Web-API in Settings → Network aktivieren"
  - MSFS: „Sim noch im Boot"
  - Fuel = 0 (Tank leer / Boot)
  - Bid-Liste leer (eingeloggt?)
  - Kein Bid passt zur aktuellen Position (mit Entfernung zum nächsten Departure-Airport)
  - flight_start hat gefailt (mit konkretem Error-Code)

### Für Fenix-Piloten

#### Aircraft-Type-Anzeige korrigiert
- Vorher: „Type ?" im Activity-Log trotz erkanntem Fenix-Profil
- Jetzt: „Type A319" / „A320" / „A321" aus dem Profile abgeleitet, wenn der Sim den ICAO-Code nicht liefert

#### Falsche Squawk-Einträge unterdrückt
- Vorher: Activity-Log loggte „Squawk set to 2000" / „Squawk set to 2532" basierend auf dem Standard-SimVar, der bei Fenix nicht mit dem RMP synchronisiert ist
- Jetzt: Bei erkanntem Fenix wird kein Squawk geloggt (`transponder_code: None`), bis wir den richtigen Fenix-eigenen LVAR identifiziert haben

### Für X-Plane-Piloten

#### Plugin-Version meldet endlich die Wahrheit
- Plugin-Version in `X-Plane Log.txt` zeigte fälschlicherweise „v0.5.0" obwohl der Code längst v0.5.13 ist
- Jetzt: Log meldet die echte Version, CMake-injected
- Wer ein altes Plugin im Community-Ordner hat: einfach das neue ZIP aus dem v0.7.17-Release rüberkopieren

### Was Piloten tun müssen

Nichts — Auto-Updater rollt v0.7.17 wie üblich aus. Optional für X-Plane-Piloten: Plugin-ZIP aus dem Release neu installieren, dann sieht der Log endlich die richtige Plugin-Version.

## 🇬🇧 English

**v0.7.17 — Fenix polish and bug bundle. Seven bugs from v0.7.16 tester feedback fixed. Highlights: Fenix profile is now auto-detected (no toggle), SimBrief refresh finally works at flight start, and Auto-Start tells you why it doesn't fire when it doesn't fire.**

### For all pilots

#### Fenix is detected automatically
- No more opt-in toggle in Settings → Beta features
- As soon as AeroACARS detects a FenixA319 / A320 / A321, the improved cockpit LVAR mappings (landing / nose / wing lights, park brake, signs etc.) kick in automatically
- If you fly Fenix: no action required, just works

#### SimBrief refresh now takes effect at flight start
- Before: clicking "Refresh" in the bid tab + starting the flight still loaded the old OFP data
- Now: flight start fetches the **most recent** OFP directly from simbrief.com (provided you have a SimBrief identifier set in Settings → SimBrief)
- Falls back to the phpVMS bid pointer if the direct fetch fails
- The "regenerate in SimBrief → Refresh → start flight" workflow now works as expected

#### Runway utilization score is finally aircraft-aware
- A321 on a 3 km runway with 2 km rollout? Before: 25/100 ("very long rollout"). Now: 80/100 ("good stop")
- Three aircraft categories with their own thresholds: Light GA / Medium narrowbody / Heavy widebody
- Cessna pilots see no change — thresholds identical to v0.7.16

#### Auto-Start: now with diagnostics
- When Auto-Start doesn't fire, the reason now shows up in the activity log:
  - X-Plane: "Aircraft title missing → enable Web API in Settings → Network"
  - MSFS: "Sim still booting"
  - Fuel = 0 (tank empty / boot)
  - Bid list empty (logged in?)
  - No bid matches current position (with distance to nearest departure airport)
  - flight_start failed (with concrete error code)

### For Fenix pilots

#### Aircraft type display fixed
- Before: "Type ?" in the activity log despite the Fenix profile being detected
- Now: "Type A319" / "A320" / "A321" derived from the profile when the sim doesn't supply the ICAO code

#### False squawk entries suppressed
- Before: activity log logged "Squawk set to 2000" / "Squawk set to 2532" based on the standard SimVar, which isn't synced with Fenix's RMP
- Now: when Fenix is detected, no squawk is logged (`transponder_code: None`) until we identify the proper Fenix-specific LVAR

### For X-Plane pilots

#### Plugin version finally reports the truth
- Plugin version in `X-Plane Log.txt` falsely showed "v0.5.0" although the code is at v0.5.13
- Now: log reports the actual version, CMake-injected
- If you have an old plugin in your community folder: copy the new ZIP from the v0.7.17 release over the top, then the log shows the correct plugin version

### What pilots need to do

Nothing — auto-updater rolls v0.7.17 out as usual. Optional for X-Plane pilots: re-install the plugin ZIP from the release, then the log finally shows the right plugin version.
